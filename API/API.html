API Testing Notes

A comprehensive guide to API testing concepts, tools, best practices, and interview preparation

What is API Testing?

API (Application Programming Interface) testing is a type of software testing that validates:

  • Endpoints, request/response
  • Business logic and data handling
  • Security and performance aspects
  • Focuses on backend testing (without UI)

API is used to communicate between two systems. It is simply known as sending the request from one system to another system and getting the required response.

Example: Communication between IRCTC and OTP. Here we send request from IRCTC App to get the response as OTP.

Typically done using tools like Postman, REST Assured, Pytest, SoapUI, Karate, JMeter, etc.

Why API Testing?
  • Early defect detection (before UI is ready)
  • Ensures functionality, reliability, performance, and security
  • Saves time in regression compared to UI automation
  • Validates integration between services
  • Language-agnostic testing approach
  • More stable than UI tests (less prone to changes)

Advantages of API:

  • API provides security through an API layer between systems
  • Helps avoid data breaching and hacking
  • Increases performance and balances load using microservices replicas
  • Helps in data hiding
  • Ensures proper communication between systems
  • Checks and authenticates the data being passed
  • Tests core functionality
  • Time effective - can hit many APIs in less time
  • Language independent (XML, JSON, HTML, TEXT, etc.)
  • Easy interaction with GUI
Types of API
  • REST API (JSON/XML, HTTP methods) - Most common
  • SOAP API (XML, WSDL-based) - More structured
  • GraphQL API (query-based) - Efficient data retrieval
  • gRPC API (high-performance, protocol buffers) - Microservices
  • WebSocket APIs - Real-time communication

API Testing Checklist
  • Verify status codes (200, 201, 400, 401, 403, 404, 500, etc.)
  • Validate response body (correct fields, values, data types)
  • Check headers (Content-Type, Authorization, CORS, etc.)
  • Authentication (Basic Auth, OAuth, JWT, API Keys)
  • Error handling (invalid input, missing params)
  • Data integrity (response matches DB / expected schema)
  • Performance (response time, load testing)
  • Security (SQL injection, XSS, token expiration)

Best Practices
  • Always test with valid and invalid inputs
  • Use environment variables (dev, QA, prod)
  • Implement data-driven testing
  • Mock APIs if backend is not ready
  • Validate both request and response payloads
  • Automate regression cases

  • Test for edge cases and boundary values

Sample API Test Cases
  • Verify GET /users returns 200 OK with correct data
  • Verify POST /login with valid creds returns token
  • Verify POST /login with invalid creds returns 401 Unauthorized
  • Verify PUT /user/{id} updates user details
  • Verify DELETE /user/{id} removes user and returns 204
  • Verify response schema matches API documentation
  • Verify API response time < 2 seconds
  • Verify pagination, sorting, filtering
  • Verify rate limiting works correctly
API Concepts

REST Concepts:

  • REST - Architecture used to create REST API
  • REST Assured - Libraries to automate REST API
  • RESTful - When we automate REST API successfully

Web Services:

  • API call over HTTP (internet) protocol is called a Web Service
  • Web service facilitates interaction between two machines over a network

WSDL (Web Service Description Language):

  • XML document containing all details about web services and API requests

UDDI (Universal Description Discovery Integration):

  • XML-based standard for describing, publishing, and finding web services

SOAP Elements:

  • Envelope - Beginning and end of message
  • Header - Header information
  • Body - Call and response information
  • Fault - Error and status information

WSDL Elements:

  • Type - Data types used by web services
  • Message - Data elements for each operation
  • Port Type - Operations that can be performed
  • Binding - Protocol and data format for each port type
SOAP vs REST Differences

REST:

  • Uses HTTP methods (GET, POST, PUT, DELETE)
  • Lightweight, usually JSON
  • Faster, more flexible
  • Example: GET /users/1

SOAP:

  • Uses XML with strict standards (WSDL)
  • Heavier, requires more bandwidth
  • Built-in security and ACID compliance
  • Example: <soap:Envelope> ... </soap:Envelope>

Key Differences:

  • API calls can be internal or over the internet, while web services call over the internet
  • Web service facilitates interaction between two machines over a network
  • API acts as an interface between two different applications
Authorization Types
  • Basic Auth - Username and password only
  • Digest Auth - More secure than Basic (username/password converted to hash)
  • Token Auth - Pass token value with Bearer keyword (most commonly used)
  • OAuth 1.0 - Consumer Key, Consumer Secret, Access Token, Token Secret
  • OAuth 2.0 - Grant Type, Client ID, Client Secret
  • No Auth - No authentication required
  • AWS Signature - Amazon Web Services signature

OAuth Usage:

  • OAuth 1.0 - Used for third-party logins
  • OAuth 2.0 - Used with single URL and different endpoints

Token authorization is mostly used in organizations because it can be easily regenerated when expired and provides good security.

Test Environment

Test Environments:

  • DEV - Where developers work
  • SIT/QA - Where testers work
  • UAT - Where testers and clients work
  • PROD - Live environment

API Test Environments:

  • Global - Large scope (variables between different collections)
  • Local - Small scope (variables from one request to another)

In QA/UAT environments, we use both Global and Local environments for API methods.

Interview Tips & Questions
Q1 Difference between API Testing and UI Testing?

Answer: API Testing focuses on backend validation â€” request/response, status codes, business logic, DB validation, and performance. It does not require a user interface.

UI Testing focuses on frontend functionality, visual elements, user interactions, and end-to-end workflows.

Example: In API testing, I validate a POST /login returns a token. In UI testing, I check if the login button works and user is redirected to the dashboard.

Q3 Tools you've used?

Answer:

  • Postman for manual API validation and creating collections
  • REST Assured (Java) for automation integrated with TestNG and Cucumber BDD
  • Pytest + Requests (Python) for lightweight automation and validations
  • Integrated API tests in Jenkins pipeline with reporting (Extent, Allure)
Q4 Authentication mechanisms (OAuth, JWT)?

Answer:

OAuth 2.0: Provides secure delegated access. Commonly used with 3rd-party logins (e.g., "Login with Google").

JWT (JSON Web Token): Compact token containing claims (user info + expiry). Passed in headers for each request (Authorization: Bearer <token>).

Example: In my projects, I handled JWT-based authentication, where I first hit the /login API, extracted the token, and used it in subsequent requests automatically.
Q5 How did you validate API responses with DB?

Answer: After API execution, I connect to the database using JDBC (Java) or PyODBC (Python). I run a SQL query to fetch the same data. I compare API response JSON fields with DB values to ensure data integrity.

Example: If /getUser/101 returns name=Pratik, I validate the same name in the Users table in DB.
Q6 How do you handle token expiry?

Answer: I check if the API returns 401 Unauthorized due to an expired token. My framework is designed to re-generate the token by calling /login or refresh token API automatically. Then, I retry the request with the new token.

This ensures my automation runs smoothly without manual intervention.
Q7 How do you integrate API tests in CI/CD?

Answer: API automation scripts are integrated into the CI/CD pipeline (Jenkins, GitHub Actions). Tests run automatically on every build/deployment. Reports (Extent/Allure/HTML) are published in Jenkins. I also set up environment variables (Dev, QA, Prod) in Jenkins to run tests in multiple environments.

This ensures continuous validation of APIs after every code change.
