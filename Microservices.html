Microservices Architecture Guide

Comprehensive overview of microservices architecture, components, and implementation

Monolith Architecture

  • Develop all functionalities in single app
  • Application will be packaged as one fat jar / fat war
  • App will be deployed in single server

Drawbacks

  1. Single Point of failure
  2. Re-Deploy entire app
  3. Maintenance of the app
  4. Burden on server

Microservices

  • Microservices is not a technology
  • Microservices is not a framework
  • Microservices is not an API
  • Microservices is an architectural design pattern
  • Microservices design pattern is universal
  • The main aim of microservices is to develop app with loosely coupling
  • Microservices based application means collection of REST APIs
  • Microservices means independently deployable and executable services

Benefits

  1. Loosely Coupled
  2. Easy Maintenance
  3. Load will be distributed
  4. Technology Independency
  5. High Availability

Challenges

  1. Bounded Context (deciding no. of REST APIs to develop)
  2. Duplicate Configuration
  3. Visibility

Microservices Architecture

There is no standard architecture for Microservices development. People are customizing microservices project architecture according to their requirement.

1. Service Registry

Maintains list of services available in the project

2. Admin Server

UI to monitor and manage all APIs using actuator endpoints

3. Zipkin Server

Used for distributed tracing

4. Backend Services

REST APIs containing business logic

5. API Gateway

Manages backend APIs and acts as entry point

6. Feign Client

Used for inter-service communication

7. Config Server

Externalizes configuration properties

8. Apache Kafka

Message broker for event-driven applications

9. Redis Cache

Used for distributed cache implementation

10. Docker

Containerization platform

Service Registry

  • Maintains list of services available in the project
  • Provides information about registered services (name, URL, status)
  • Provides number of instances available for each service
  • We can use Eureka Server as a service registry
  • Eureka server provided by Spring Cloud Netflix library

Steps to develop Service Registry Application (Eureka Server)

1. Create Service Registry application with EurekaServer dependency

<!-- pom.xml -->
<dependency>
  <groupId>org.springframework.cloud</groupId>
  <artifactId>spring-cloud-starter-netflix-eureka-server</artifactId>
</dependency>

2. Configure @EnableEurekaServer annotation in boot start class

// Application.java
@SpringBootApplication
@EnableEurekaServer
public class Application {
  public static void main(String[] args) {
    SpringApplication.run(Application.class, args);
  }
}

3. Configure properties in application.yml file

# application.yml
server:
  port: 8761

eureka:
  client:
    register-with-eureka: false

If Service-Registry project port is 8761 then clients can discover service-registry and will register automatically.

4. Access Eureka Dashboard

URL: http://localhost:8761/

Admin Server

  • Actuators are used to monitor and manage our applications
  • Monitoring and managing all the APIs separately is challenging
  • Admin Server provides a UI to monitor and manage all APIs at one place using actuator endpoints

Steps to develop Spring Admin-Server

  1. Create Boot application with admin-server dependency
  2. Configure @EnableAdminServer annotation at start class
  3. Change Port Number (Optional)
  4. Run the boot application
  5. Access application URL in browser

Zipkin Server

  • Used for Distributed tracing
  • Monitor which API is taking more time to process request
  • Understand how many APIs involved in request processing

Steps to work with Zipkin Server

  1. Download Zipkin Jar file from https://zipkin.io/pages/quickstart.html
  2. Run zipkin jar file: java -jar <jar-name>
  3. Zipkin Server runs on port 9411
  4. Access dashboard: http://localhost:9411/

Backend APIs

  • Contain business logic
  • Also called as REST APIs / services / microservices
  • Examples: payment-api, cart-api, flights-api, hotels-api
  • Can register as client for Service Registry, Admin server & Zipkin server (optional)

Feign Client

  • Provided by Spring Cloud libraries
  • Used for Inter Service Communication
  • One API accessing another API using Service Registry
  • No need to mention URL of the API to access
  • Uses service name to get service URL from service registry
  • Uses Ribbon for client-side load balancing

API Gateway

  • Used to manage backend APIs
  • Acts as mediator between user requests and backend APIs
  • Acts as entrypoint for all backend APIs
  • Contains two types of logics:
    1. Request Filter: To validate the request (go / no-go)
    2. Request Router: Forward request to particular backend-API based on URL pattern

API Gateway Configuration

# application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: welcome-api
          uri: lb://WELCOME-API
          predicates:
            - Path=/welcome
        - id: greet-api
          uri: lb://GREET-API
          predicates:
            - Path=/greet

Config Server

  • Part of Spring Cloud Library
  • Used to externalize config properties of application
  • In realtime, config properties are kept outside of the project to simplify maintenance
  • Config properties files are maintained in GitHub repo
  • Config server loads properties and provides them to applications based on application name

Apache Kafka

  • Message broker
  • Works based on Publisher-Subscriber model
  • Sends messages from one app to another app as a mediator
  • Used to develop Event Driven Microservices based applications

Redis Cache

  • Two types of tables in applications:
    1. Transactional tables (DML operations)
    2. Non-Transactional tables (DQL operations)
  • Not recommended to load non-transactional tables data from DB every time
  • Cache reduces number of round trips between Java app and Database
  • Redis is used for distributed cache implementation

Steps to develop WELCOME-API

  1. Create Spring Boot application with dependencies:
    • eureka-discovery-client
    • starter-web
    • devtools
    • actuator
    • zipkin
    • admin-client
  2. Configure @EnableDiscoveryClient annotation at boot start class
  3. Create RestController with required method
  4. Configure properties in application.yml file:
    # application.yml
    server:
      port: 1111
    spring:
      application:
        name: WELCOME-API
      boot:
        admin:
          client:
            url: http://localhost:9090/
    eureka:
      client:
        serviceUrl:
          defaultZone: http://localhost:8761/eureka
    management:
      endpoints:
        web:
          exposure:
            include: '*'
  5. Run the application and check in Eureka Dashboard
  6. Check Admin Server Dashboard
  7. Send Request to REST API method
  8. Check Zipkin Server UI

Inter-service Communication

  1. Add @EnableFeignClients dependency in GREET-API boot start class
  2. Create FeignClient interface:
    // WelcomeApiClient.java
    @FeignClient(name = "WELCOME-API")
    public interface WelcomeApiClient {
      @GetMapping("/welcome")
      public String invokeWelcomeMsg();
    }
  3. Inject feign client into GreetRestController:
    // GreetRestController.java
    @RestController
    public class GreetRestController {
      @Autowired
      private WelcomeApiClient welcomeClient;

      @GetMapping("/greet")
      public String getGreetMsg() {
        String welcomeMsg = welcomeClient.invokeWelcomeMsg();
        String greetMsg = "Good Morning, ";
        return greetMsg.concat(welcomeMsg);
      }
    }
  4. Run the applications and access greet-api method

Load Balancing

  • Distribute requests to multiple servers
  • Run welcome-api in multiple instances:
    1. Remove port number configuration from welcome-api yml file
    2. Make changes in rest controller to display port number in response
    3. Run with different ports using VM arguments:
      -Dserver.port=8081
      -Dserver.port=8082

Auto Scaling

  • Scale up or scale down servers based on incoming traffic
  • Benefits:
    1. Fault Tolerance
    2. High Availability
    3. Cost Management

Circuit Breaker Design Pattern

  • Protects from high voltage or low voltage power (electrical concept)
  • Diverts traffic when problems detected in normal execution flow
  • Used in microservices to implement fault tolerance/resilience systems
  • When main logic fails continuously, executes fallback logic for some time

Circuit Breaker Implementation

  1. Create Spring Boot project with dependencies:
    • web-starter
    • actuator
    • aop
    • resilience4j
  2. Create Rest Controller:
    // DataRestController.java
    @RestController
    public class DataRestController {
      @GetMapping("/data")
      @CircuitBreaker(fallbackMethod = "getDataFromDB", name = "ashokit")
      public String getData() {
        System.out.println("redis method called..");
        int i = 10 / 0;
        return "Redis Data sent to u r email";
      }

      public String getDataFromDB(Throwable t) {
        System.out.println("db method called..");
        return "DB Data sent to u r email";
      }
    }
  3. Configure Circuit Breaker Properties:
    # application.yml
    spring:
      application.name: resilience4j-demo
    management:
      endpoints.web.exposure.include: '*'
      endpoint.health.show-details: always
      health.circuitbreakers.enabled: true
    resilience4j.circuitbreaker:
      configs:
        default:
          registerHealthIndicator: true
          slidingWindowSize: 10
          minimumNumberOfCalls: 5
          permittedNumberOfCallsInHalfOpenState: 3
          automaticTransitionFromOpenToHalfOpenEnabled: true
          waitDurationInOpenState: 5s
          failureRateThreshold: 50
          eventConsumerBufferSize: 10
  4. Test the application and monitor actuator health endpoint
