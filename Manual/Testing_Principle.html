Seven Principles of Software Testing

Understanding these fundamental principles helps improve testing effectiveness and efficiency in software development.

1

Testing shows presence of defects

Testing can prove that bugs exist, but it cannot guarantee that the software is completely bug-free.

Testing shows presence of defects, not their absence

Testing can prove that bugs exist, but it cannot guarantee that the software is completely bug-free. No matter how much testing you perform, you can never be 100% certain that there are no defects remaining. Testing reduces the probability of undiscovered defects but doesn't eliminate the possibility entirely.

2

Exhaustive testing is impossible

You cannot test all inputs, paths, and scenarios. Instead, focus on risk-based and prioritized testing.

Exhaustive testing is impossible

You cannot test all inputs, paths, and scenarios. Instead, focus on risk-based and prioritized testing. With complex software systems, the number of possible test scenarios is virtually infinite. Smart testing involves analyzing risks, prioritizing test cases based on impact and likelihood, and using techniques like equivalence partitioning and boundary value analysis to maximize test coverage efficiently.

3

Early testing saves time and money

The earlier you start testing, the cheaper it is to fix defects.

Early testing saves time and money

The earlier you start testing (e.g., requirement/design phase), the cheaper it is to fix defects. Defects found and fixed early in the development lifecycle are significantly less expensive to resolve than those discovered later. Early testing activities include reviewing requirements, design documents, and code, which can identify issues before they become embedded in the software.

4

Defects cluster together

A small number of modules contain most of the defects (Pareto principle: 80/20 rule).

Defects cluster together

A small number of modules contain most of the defects (Pareto principle: 80/20 rule). This principle suggests that defects are not evenly distributed throughout the software but tend to cluster in specific components or modules. Identifying these high-risk areas through defect analysis and metrics allows testers to focus their efforts where they're most needed.

5

Pesticide paradox

Repeating the same tests will no longer find new defects. Tests need to be reviewed and updated regularly.

Pesticide paradox

Repeating the same tests again and again will no longer find new defects. Tests need to be reviewed and updated regularly. Just as insects develop resistance to pesticides, software becomes "immune" to repeated test cases. To combat this, test suites must be regularly reviewed, updated, and expanded with new test cases and techniques to uncover different types of defects.

6

Testing is context dependent

Testing approach varies depending on the domain and application type.

Testing is context dependent

Testing approach varies depending on the domain (e.g., banking requires security testing, e-commerce requires usability & performance testing). There is no one-size-fits-all testing approach. The testing strategy, techniques, and focus areas should be tailored to the specific context, including the technology, business domain, risk profile, and regulatory requirements of the software.

7

Absence-of-errors fallacy

Just because software is bug-free doesn't mean it's useful or meets business needs.

Absence-of-errors fallacy

Just because software is bug-free doesn't mean it's useful. The software must meet business needs and requirements. Finding and fixing defects is important, but it's not the ultimate goal. The real measure of software quality is whether it satisfies user needs and provides value. A defect-free application that doesn't solve the right problem or meet user expectations is still a failure.
