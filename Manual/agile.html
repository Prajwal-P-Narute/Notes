What is Agile Methodology?

Agile Methodology is an iterative approach to software development that focuses on delivering value to customers through incremental releases, collaboration, and adaptability to change.

Definition

Agile is a time-boxed, iterative approach to software delivery that builds software incrementally from the start of the project, instead of trying to deliver it all at once near the end. It works by breaking projects down into little bits of user functionality called user stories, prioritizing them, and then continuously delivering them in short two-week cycles called iterations.

Key Principles of Agile

Work in Sprints

Projects are divided into smaller iterations (sprints) for regular delivery

Customer Involvement

Continuous customer feedback ensures product meets expectations

Adaptability

Easy accommodation of changing requirements

Team Collaboration

Cross-functional teams working together

Continuous Improvement

Regular retrospectives for process refinement

Popular Agile Frameworks

Scrum

Teams work in time-boxed sprints with roles like Scrum Master and Product Owner

Kanban

Visual workflow management using boards to limit work in progress

XP (Extreme Programming)

Emphasizes technical excellence with practices like pair programming and TDD

Agile Roles & Responsibilities

Stakeholder

Customer or representative from the top management. Provides business requirements and priorities. Can change requirements at any phase. Provides a bundle of requirements to the Product Owner.

Product Owner

Similar to Business Analyst in traditional models. Gathers all requirements from stakeholders. Creates and maintains the Product Backlog. Participates in grooming and sprint planning meetings. Prioritizes user stories for development.

Scrum Master

Facilitates the Scrum process. Removes impediments for the team. Ensures adherence to Agile principles. Conducts sprint planning and retrospective meetings. Protects the team from external distractions.

Agile Artifacts

Product Backlog

Similar to Business Requirements Specification (BRS). Contains all requirements for the entire project. Prioritized by the Product Owner. Continuously refined and updated.

Sprint Backlog

Similar to System Requirements Specification (SRS). Contains user stories selected for the current sprint. Provides detailed information for module development. Owned by the Development Team.

User Stories

Functional requirements for module development. Written in format: "As a [user], I want [feature] so that [benefit]". Include acceptance criteria for testing. Estimated using story points or ideal days.

Agile Meetings & Ceremonies

Sprint Planning

Duration: 1-2 hours

Attendees: Scrum Master, Development Team, Testing Team, Product Owner

Purpose: Job allocation, resource allocation, and estimation

Backlog Grooming

Duration: 1-2 hours (weekly or bi-weekly)

Attendees: Scrum Master, Development Team, Testing Team, Product Owner

Purpose: Review, refine, and prioritize the product backlog

Activities:

  • Break down large user stories
  • Write and estimate user stories
  • Prioritize backlog items
  • Clarify requirements and acceptance criteria

Daily Stand-up

Duration: 15-30 minutes daily

Attendees: Scrum Master, Development Team, Testing Team, Product Owner

Purpose: Progress tracking and impediment identification

Three questions:

  • What did I do yesterday?
  • What will I do today?
  • What obstacles are in my way?

Sprint Review

Duration: 1-2 hours

Attendees: Scrum Master, Development Team, Testing Team, Product Owner, Stakeholders

Purpose: Demo of completed work to stakeholders for feedback

Sprint Retrospective

Duration: 30-45 minutes

Attendees: Scrum Master, Development Team, Testing Team

Purpose: Process improvement discussion

Three questions:

  • What went well?
  • What could be improved?
  • What actions will we implement?

Estimation in Agile

Estimation is the process of prioritizing and sizing user stories for specific modules. It considers:

Knowledge - Team members' understanding of the project domain

Effort - Resources and manpower required to complete the work

Complexity - Time, cost, and resource requirements based on work complexity

Estimation Techniques:

  • Planning Poker: Team members use numbered cards to estimate effort
  • T-Shirt Sizing: XS, S, M, L, XL to categorize story size
  • Story Points: Relative measurement of effort (Fibonacci sequence)
  • Ideal Days: Estimating based on uninterrupted work days

Agile Advantages & Disadvantages

Advantages

  • Faster delivery of features
  • Higher customer satisfaction through regular updates
  • Improved teamwork and communication
  • Fewer defects due to frequent testing
  • Flexibility to accommodate changing requirements
  • Early and predictable delivery
  • Better risk management
  • Continuous improvement

Disadvantages

  • Requires deep understanding of software flows and dependencies
  • Limited time for knowledge transfer due to short sprints
  • Can be difficult to measure progress without clear milestones
  • May require significant cultural change in organizations
  • Can be challenging with distributed teams
  • Risk of scope creep if not properly managed
  • Documentation may be less comprehensive

Check Points in Agile

Check points are systematic processes for defect detection and prevention:

1. Defect Identification - When a defect is found in production

2. Check Point Analysis - Tester examines check points between development phases

3. Defect Localization - Using tools like AVAS to highlight defective code

4. Rapid Resolution - Developer fixes the highlighted code immediately

Benefits of Check Points:

  • Eliminates need for post-mortem testing
  • Speeds up defect resolution
  • Provides clear visibility into code issues
  • Reduces debugging time

Sanity & Smoke Testing in Agile

Sanity and Smoke testing are both types of preliminary testing performed to ensure that the basic functionalities of an application work correctly before proceeding with more comprehensive testing.

Sanity Testing

Also known as zero level testing or build verification testing

Purpose

The aim of sanity testing is to check basic or core functionality of the application to determine if it's stable enough for more detailed testing.

When Performed

Sanity testing is performed after integration testing ends and before more comprehensive testing begins.

Process

Database analyst team creates a test environment with a URL to run the build. Testers verify if the build generates desired output. Testing focuses on basic functionality without deep validation.

Test Focus Areas

  • Basic Core Functionality: Testing buttons, icons, and forms that allow users to proceed to next stages. Example: Testing the submit button on Facebook's sign-up page
  • Input Field Validation: Testing input fields to ensure they accept values correctly. Example: Verifying that text fields in Facebook accept characters, symbols, and numbers from both physical and on-screen keyboards
  • Link Validation: Checking that interlinked pages work in the correct sequence. Example: Verifying payment process links in Amazon work correctly with UPI or third-party applications
  • Page Validation: Testing navigation between pages using next and back buttons. Example: Verifying page navigation in WPS PDF reader works correctly
  • GUI Validation: Checking the visual interface where users interact directly. Example: Verifying pages display correctly, images are clear, and no blurring occurs

Smoke Testing

Advanced version of sanity testing with additional validation

Purpose

To verify that basic and critical (high priority) functionality works properly at the initial stage of a build.

Participants

Both back-end developers and testers are involved in smoke testing.

Formula

Smoke Test = Sanity Test + Troubleshooting + Package Validation

Package Validation

Package is a collection of objects downloaded to the system when installing a build. Smoke testing validates that all required packages are correctly installed and functional.

Primary Goal

To ensure the software build is stable enough for further testing.

Difference Between Smoke and Sanity Testing

Criteria Smoke Testing Sanity Testing
Definition Build verification testing conducted to verify basic and critical functionality Zero level testing conducted to verify basic and core functionality
When Performed Initial stage of the build After build passes smoke testing
Scope Combination of sanity testing, troubleshooting, and package validation Done on enhancement features; kind of regression testing
Primary Goal Ensure software build is stable for further testing Ensure build is ready for detailed level testing
Performed By Developers and testers QA team
Depth of Testing More comprehensive, includes package validation Surface level, checks only basic functionality
Documentation Usually documented with test cases Often undocumented, exploratory in nature

Benefits of Early Stage Testing

  • Early Defect Detection: Identify critical issues before detailed testing begins
  • Cost Efficiency: Reduce cost by finding issues early in the development cycle
  • Time Savings: Avoid wasting time on detailed testing of unstable builds
  • Better Resource Allocation: Focus testing efforts on builds that pass initial verification

Best Practices for Sanity and Smoke Testing

  1. Create a Checklist: Develop a standardized checklist of critical functionalities to verify in each build
  2. Automate When Possible: Automate repetitive smoke tests to save time and ensure consistency
  3. Prioritize Critical Paths: Focus on the most critical user journeys that would block further testing if broken
  4. Maintain Documentation: Keep records of test results to identify patterns of recurring issues
