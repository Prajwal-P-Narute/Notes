Software Testing Manual

Comprehensive guide to software testing concepts, methodologies, and best practices

Introduction to Software Testing

Software is the collection of specialized programs which take user input & generate desired output.

Software testing is the process of verifying that a given software or application is generating the desired output or not. It is the process of checking completeness and correctness of the software with respect to customer expectations.

]

Resources involved in Software Development

  • Customer - Provides requirements and expectations
  • Business Analyst - Gathers and documents requirements
  • Developer - Develops application as per requirements
  • Tester - Checks correctness & completeness of functionality
  • End User - Ultimately uses the final product

Software Quality Assurance (SQA)

SQA is communication between customer & BA. It ensures the product quality by considering several factors:

Meeting Requirements

Identify type of application (banking, telecom, healthcare, finance, etc.) and ensure it meets all specified requirements.

Meeting Expectations

Ensure privacy and security of the application, and verify performance under heavy load conditions.

Cost Management

Manage project costs as per customer requirements and available resources.

Timely Delivery

Deliver the project within the agreed timeframe to avoid penalties (escalation).

Maintenance

Provide service after delivery and fix any post-delivery issues free of cost.

Key SQA Factors

  • To meet customers' requirements - Identify type of application (banking, telecom, healthcare, finance, etc.)
  • To meet customers' expectations
    • Privacy → includes security of application
    • Performance → application should be stable under heavy load
  • Cost of project → Decided as per customer requirements & resources
  • Timing of delivery → Discussed during requirement gathering. Delay may cause penalty (escalation)
  • Maintenance → Service provided after delivery. Any post-delivery issues must be fixed free of cost

Software Development Life Cycle (SDLC)

Software development life cycle consists of set of particular stages which start from information gathering to the maintenance of product.

Stages in SDLC

  1. Information gathering
  2. Analysis
  3. Design
  4. Coding (developer)
  5. Testing
  6. Maintenance

1. Information Gathering

  • BA is responsible for gathering the information from customers
  • Information gathering is nothing but collecting requirement from customers
  • Information gathering involves business requirements specification (BRS)
  • BRS is bridge between client to developers & testers
  • BRS is a document that contains all (overall) requirements related to the application
  • BA is responsible for preparing BRS documents

2. Analysis

  • Here also BA is responsible person
  • In this phase BA will prepare software requirement specification (SRS)
  • SRS is detail level documents
  • SRS document is also known as functional requirement specification

SRS Components:

  • Functional flow diagram
    • It is flow of our application
    • This flow shows relation between the tasks
    • It will give the idea about sequence of functionality
    • Relationship of functionality is nothing but dependency of each other
    • Ex (sign up page → login page → home page → account page)
  • Functional requirements
    • It is nothing but attributes which are required to complete a specific function
    • Ex. Sign up page - for sign up page required 1st name, last name, mob no, email id, passwords, sign up button
    • Ex. For name should be some requirements
      1. Name should be in character only
      2. Name should not accept number
      3. It should not accept any blank space
      4. It should not accept any special symbols
  • Use cases
    • Use cases is nothing but functionality in term of input and output
    • Use cases is nothing but users how to deal with functionality to achieve target
    • It help to create foundation for creating user stories or test cases in agile methodologies
  • Snapshots
    • Snapshots are visualization of any functionality before development of any application
    • Snapshot are created by BA
    • Snapshot give idea to developer and designer that how software supposed to look like

3. Design

High Level Design

  • High level design contains the working of main module
  • It includes relation and dependency of main module
  • It includes what and how any module works
  • High level design is created by design Architecture

Low Level Design

  • It designs contain the working of sub modules
  • Low level design is created by Front end developer
  • Ex. If any sign-up page is main module, then name, password, mobile no etc. are the sub functionality of main module

4. Coding

  • It means developing the codes
  • One line is code
  • Multiple line of code is program
  • Set of programs written by developer creates the software

Code is written by three types of developer

  1. Front end developer
    • UI functionality, function flows, process are developed by the front-end developer
  2. Back-end developer

  3. Full stack developer
    • Note: person/developer who can work as front end as well as back-end developer known as full stack developer

5. Testing

Testing is the process checking a completeness and correctness of the software.

Types of Testing

White Box Testing Black Box Testing
It is also called as Unit code level testing, and clear box testing Black box testing is also known system and function testing
Developer is responsible for white box testing Tester is responsible for black box testing
Developer will debug or compile the code and check the code if any error is there Tester has to check overall functionality (internal and external) of the application
Developer will test only positive scenario Tester has to test app in positive as well as negative way

Software Development Process Models

Fish Model

The Fish Model is an advanced version of the SDLC that consists of verification and validation processes working together to ensure software quality.

Verification vs Validation

Verification Validation
Verification is the process of checking whether the developed product fulfills the customer's requirements or not Validation is the process of checking whether the developed product is free from bugs or not
Verification is done in different stages of SDLC Validation is done after complete development and at the testing stage
It is also known as static testing It is also known as Quality control or dynamic testing
The focus of verification is on methods/process like reviews, inspection and walk through etc. The focus of validation is on Quality of software, so we can deliver software free from bugs
It helps to find the bug in early stage It helps to increase the quality of software
Quality assurance team does the verification Quality analyst team does the validation
Budget required less for verification as compared to validation Budget required for validation is more than verification

Waterfall Model

  • Waterfall model is step by step implementation of SDLC
  • In waterfall model requirements are fixed
  • If first stage is complete and process goes to next phase, then we cannot revert back to the first phase
  • Generally, waterfall model is used in small scale company or small application
  • The general time duration of waterfall model is 3 months
  • In waterfall model, when tester finds any bug then they do not send that back to the developer, instead they log the bug, make report & those bugs get fixed in next version of software

V-Model

  • V stands for verification and validation
  • V model has working in which verification and validation phases run parallel
  • In V model development stages are mapped with testing stages
  • In V model, if one stage gets completed and 2nd is running but changes occurred in 1st phase then we can revert back to 1st phase but customer have to pay some amount to company
  • The duration of V model is 3 months
  • It is used for large application. Generally, this method is not used nowadays

Agile Methodology

Agile Methodology is an iterative approach to software development that focuses on delivering value to customers through incremental releases, collaboration, and adaptability to change.

Key Principles of Agile

  • Work in Sprints: Projects are divided into smaller iterations (sprints) for regular delivery
  • Customer Involvement: Continuous customer feedback ensures product meets expectations
  • Adaptability: Easy accommodation of changing requirements
  • Team Collaboration: Cross-functional teams working together
  • Continuous Improvement: Regular retrospectives for process refinement

Scrum

Teams work in time-boxed sprints with roles like Scrum Master and Product Owner

Kanban

Visual workflow management using boards to limit work in progress

XP (Extreme Programming)

Emphasizes technical excellence with practices like pair programming and TDD

Stakeholder

  • Similar to Business Analyst in traditional models
  • Gathers all requirements from stakeholders
  • Creates and maintains the Product Backlog

Scrum Master

  • Facilitates the Scrum process
  • Removes impediments for the team
  • Ensures adherence to Agile principles
  • Conducts sprint planning and retrospective meetings
  • Protects the team from external distractions

Agile Artifacts

Product Backlog

  • Similar to Business Requirements Specification (BRS)
  • Contains all requirements for the entire project
  • Prioritized by the Product Owner
  • Continuously refined and updated

Sprint Backlog

  • Similar to System Requirements Specification (SRS)
  • Contains user stories selected for the current sprint
  • Provides detailed information for module development
  • Owned by the Development Team

User Stories

  • Functional requirements for module development
  • Written in format: "As a [user], I want [feature] so that [benefit]"
  • Include acceptance criteria for testing
  • Estimated using story points or ideal days

Attendees: Scrum Master, Development Team, Testing Team, Product Owner

Duration: 1-2 hours

Attendees: Scrum Master, Development Team, Testing Team, Product Owner, Stakeholders

Duration: 30-45 minutes

Attendees: Scrum Master, Development Team, Testing Team

Purpose: Process improvement discussion

Three questions:

Estimation in Agile

Estimation is the process of prioritizing and sizing user stories for specific modules. It considers:

Knowledge

Team members' understanding of the project domain

Effort

Resources and manpower required to complete the work

Complexity

Time, cost, and resource requirements based on work complexity

Estimation Techniques:

  • Planning Poker: Team members use numbered cards to estimate effort
  • T-Shirt Sizing: XS, S, M, L, XL to categorize story size
  • Story Points: Relative measurement of effort (Fibonacci sequence)
  • Ideal Days: Estimating based on uninterrupted work days

Sanity & Smoke Testing

Sanity and Smoke testing are both types of preliminary testing performed to ensure that the basic functionalities of an application work correctly before proceeding with more comprehensive testing.

Sanity Testing

Also known as zero level testing or build verification testing

Purpose

The aim of sanity testing is to check basic or core functionality of the application to determine if it's stable enough for more detailed testing.

When Performed

Sanity testing is performed after integration testing ends and before more comprehensive testing begins.

Process

  1. Database analyst team creates a test environment with a URL to run the build
  2. Testers verify if the build generates desired output
  3. Testing focuses on basic functionality without deep validation

Test Focus Areas

Basic Core Functionality

Testing buttons, icons, and forms that allow users to proceed to next stages

Example: Testing the submit button on Facebook's sign-up page
Input Field Validation

Testing input fields to ensure they accept values correctly

Example: Verifying that text fields in Facebook accept characters, symbols, and numbers from both physical and on-screen keyboards
Link Validation

Checking that interlinked pages work in the correct sequence

Example: Verifying payment process links in Amazon work correctly with UPI or third-party applications
Page Validation

Testing navigation between pages using next and back buttons

Example: Verifying page navigation in WPS PDF reader works correctly
GUI Validation

Checking the visual interface where users interact directly

Example: Verifying pages display correctly, images are clear, and no blurring occurs

Smoke Testing

Advanced version of sanity testing with additional validation

Purpose

To verify that basic and critical (high priority) functionality works properly at the initial stage of a build.

Participants

Both back-end developers and testers are involved in smoke testing.

Formula

Smoke Test = Sanity Test + Troubleshooting + Package Validation

Package Validation

Package is a collection of objects downloaded to the system when installing a build. Smoke testing validates that all required packages are correctly installed and functional.

Primary Goal

To ensure the software build is stable enough for further testing.

Difference Between Smoke and Sanity Testing

Criteria Smoke Testing Sanity Testing
Definition Build verification testing conducted to verify basic and critical functionality Zero level testing conducted to verify basic and core functionality
When Performed Initial stage of the build After build passes smoke testing
Scope Combination of sanity testing, troubleshooting, and package validation Done on enhancement features; kind of regression testing
Primary Goal Ensure software build is stable for further testing Ensure build is ready for detailed level testing
Performed By Developers and testers QA team
Depth of Testing More comprehensive, includes package validation Surface level, checks only basic functionality
Documentation Usually documented with test cases Often undocumented, exploratory in nature

Benefits of Early Stage Testing

Early Defect Detection

Identify critical issues before detailed testing begins

Cost Efficiency

Reduce cost by finding issues early in the development cycle

Time Savings

Avoid wasting time on detailed testing of unstable builds

Better Resource Allocation

Focus testing efforts on builds that pass initial verification

2

3

Focus on the most critical user journeys that would block further testing if broken

4

Maintain Documentation

Keep records of test results to identify patterns of recurring issues

Integration Testing

Integration is the process of mapping new modules with existing modules. Integration Testing is the process of verifying the interaction and data flow between interconnected modules/components of an application.

Main Purpose of Integration Testing

Focus on validating data flow, error handling, and module interactions.

Data Flow Validation

Ensure data passes correctly between modules

Error Handling

Verify proper error handling across module boundaries

Module Interactions

Test interfaces and interactions between modules

Types of Integration Testing

Big Bang Approach

All modules are integrated and tested together.

Pros
  • Simple
  • Suitable for small systems
Cons
  • Difficult to catch defects
  • Debugging is challenging
Incremental Approach

Modules are integrated and tested step by step.

Top-Down

Starts with high-level modules and moves to lower-level ones

Bottom-Up

Starts with lower-level modules and progresses upward

Hybrid/Sandwich

Combines both top-down and bottom-up approaches

Stub and Driver-Based

Used when modules are dependent but incomplete.

Stub

A dummy implementation for lower-level module that provides responses to the higher-level module

Driver

A dummy implementation for higher-level module that supplies inputs to the lower-level module

Black Box Testing

Black box testing, also known as system integration testing, functional testing, or detail level testing, is a software testing method where the internal structure/design/implementation of the item being tested is not known to the tester. The tester only focuses on the input and output of the software system without knowing how the system processes the input to produce the output.

Tester Involvement

Testers are primarily involved in this testing approach, focusing on the external behavior of the system.

Testing Scope

Involves checking the overall functionality including both internal and external aspects of the application.

System & Function Testing

Includes functionality testing (high involvement), usability testing (high involvement), security testing (less involvement), and performance testing (less involvement).

Functionality Testing

Functionality testing is the process of checking correctness and completeness of functionality. It ensures that the software application functions according to the specified requirements.

Functional Testing

Focuses on the internal functionality of the application including:

  • Behavioral coverage
  • Input domain coverage
  • Error handling coverage
  • Back-end coverage
  • Service level coverage
  • Calculation base coverage

Non-functional Testing

Focuses on the external functionality and quality attributes of the application including:

  • Recovery testing
  • Compatibility testing
  • Configuration testing
  • Inter system testing
  • Installation testing
  • Globalization testing
  • Gold Plating
  • Parallel testing

Functional Testing Coverage Details

1. Behavioral Coverage

In behavioral coverage, testers check the property and behavior of objects within the application.

Example:

In a bus booking application, when a user hovers over a seat radio button, it gets highlighted with a bright color (behavior). After clicking, the seat gets booked (property).

2. Input Domain Coverage

This coverage focuses on validating the size and data type of inputs using techniques like Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP).

Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP)

Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP) are two widely used techniques in software testing to identify test cases efficiently.

Equivalence Class Partitioning (ECP)

What it does: It divides input data into groups or "partitions" where all inputs in the same group are expected to behave similarly.

Why use it: To reduce the total number of test cases while still covering maximum functionality.

How it works:
  1. Identify all possible inputs for a feature or function.
  2. Group these inputs into "valid" and "invalid" partitions.
  3. Pick one representative value from each partition to test.
Example:

For a login form requiring a password of 8 to 12 characters:

  • Valid Partition: Inputs with 8 to 12 characters (e.g., "password123")
  • Invalid Partitions:
    • Less than 8 characters (e.g., "pass")
    • More than 12 characters (e.g., "passwordislonger")

Boundary Value Analysis (BVA)

What it does: Focuses on testing values at the "edges" of input ranges because errors are more likely there.

Why use it: To catch boundary-related defects.

How it works:
  1. Identify the boundaries of input ranges (minimum, maximum, and just outside these limits).
  2. Test values on, just below, and just above these boundaries.
Example:

Using the same password example (8 to 12 characters):

  • Boundaries:
    • Minimum: 8 characters
    • Maximum: 12 characters
  • Test cases:
    • On the boundary: 8, 12 characters (e.g., "password", "pass123456")
    • Just outside: 7, 13 characters (e.g., "pass123", "passwordplus")

Summary

ECP

Reduces test cases by grouping similar inputs into partitions and testing one representative value.

BVA

Focuses on edge cases at the boundaries of input ranges for more robust testing.

3. Error Handling Coverage

This coverage ensures the system displays appropriate error messages when invalid inputs or actions occur.

Example:

When a user enters only 2 digits in a 10-digit mobile number field, the system should display an error message like "Please enter a 10-digit mobile number" and highlight the field in red.

4. Back-end Coverage

This coverage verifies that data is correctly stored in and retrieved from the database.

Example:

In a financial app, all personal information provided during account opening (name, mobile number, PAN, etc.) should be correctly stored in the database.

5. Service Level Coverage

This coverage ensures the system works according to the function flow diagram created by business analysts.

Example:

In a cloud storage app, the sequence of pages (home page → folders → saved data) should follow the predefined flow without deviations.

6. Calculation Base Coverage

This coverage validates that arithmetic operations within the application work correctly.

Example:

In an e-commerce app, purchasing 2 items priced at ₹1500 each should correctly calculate the total as ₹3000. Adding 2 more items should update the total to ₹6000.

Non-functional Testing Types

1. Recovery Testing / Reliability Testing

Process of checking whether an application can recover from abnormal situations to normal operation.

Example:

A result website that becomes unresponsive under heavy load should return to normal operation when the load decreases.

2. Compatibility Testing

Ensures that a software application works correctly across different environments, including browsers, operating systems, devices, networks, and hardware configurations.

Forward vs. Backward Compatibility
Criteria Forward Compatibility Backward Compatibility
Definition Tests software on newer versions Tests software on older versions
Purpose Ensures app works with future updates Ensures app works with legacy systems
Example Checking an Android 12 app on Android 14 Checking a Word 2024 file in Word 2016

3. Configuration Testing

Focuses on how the software performs with different system configurations (hardware, OS, network settings).

Example:

Testing an application on Windows with different RAM configurations (4GB, 8GB, 16GB) to ensure proper functionality across all setups.

4. Inter System Testing

Validates the interaction and data flow between multiple integrated systems to ensure they communicate correctly.

Example:

Testing how a CRM system integrates with an email marketing platform to ensure customer data syncs correctly between both systems.

5. Installation Testing

Verifies if the application installs correctly on the target platform and uninstalls without leaving residual files.

Key Checks:
  • Application creates desktop shortcuts after installation
  • All required files, packages, and drivers are present
  • User interface is intuitive and easy to navigate
  • Sufficient disk space validation before installation
  • Proper uninstallation without residual files

6. Globalization Testing

Verifies whether an application supports different languages, regions, and cultural conventions without requiring code changes.

Example:

A news app should correctly display content in different languages, maintain proper formatting for numbers, dates, and currency symbols, and support right-to-left text for languages like Arabic.

Key Aspects:
  • Text translation while keeping numbers in English format
  • Correct date, time, currency, and number formats for different regions
  • Text expansion handling when translated to different languages
  • UI adaptability including alignment, font support, and RTL text handling

7. Gold Plating

Testing of extra features not mentioned in customer requirements.

Important:

If developers add extra features with customer permission (and potential additional payment), testers do not log defects. If added without permission, testers should log these as defects.

Benefits of Black Box Testing

Independent Testing

Testers don't need knowledge of internal code, preventing developer bias

User Perspective

Tests from the end-user's viewpoint, ensuring real-world usability

Early Detection

Helps identify specification gaps and ambiguities early in the cycle

Efficient for Changes

Code changes don't affect test cases, making maintenance easier

Usability Testing

Usability testing is a type of testing where real users are asked to interact with the application to check how easy, efficient, and user-friendly it is. The main goal is to identify any problems in navigation, design, or interaction so that the product delivers a smooth user experience.

📌 Detailed Notes on Usability Testing

🔹 Objectives

  • Verify ease of use of the application.
  • Measure user satisfaction.
  • Identify navigation/design issues before release.

🔹 When to Perform

  • During the design stage (prototypes/wireframes).
  • Before releasing a new feature.
  • Before the final product launch.

🔹 Steps

  1. Plan → Define goals, tasks, and user scenarios.
  2. Recruit Users → Select real or representative users.
  3. Execute Test → Ask users to perform tasks while being observed.
  4. Collect Data → Record time taken, errors, feedback.
  5. Analyze & Report → Suggest improvements.

🔹 Types of Usability Testing

  • Remote vs. In-person – Testing done online or face-to-face.
  • Moderated vs. Unmoderated – With or without a facilitator guiding the user.
  • Explorative – Early-stage testing for ideas.
  • Comparative – Comparing two designs/versions.

🔹 Example

In a food delivery app, if users take too long to find the "Order History" button, it's a usability issue.

Performance Testing

Performance Testing is a type of software testing that ensures software applications perform well under their expected workload. It focuses on measuring a system's quality attributes like responsiveness, stability, scalability, and resource usage.

Purpose of Performance Testing

  • Evaluate system performance under various load conditions
  • Identify and eliminate performance bottlenecks
  • Ensure application meets performance requirements
  • Validate system reliability and stability
  • Measure system capacity and scalability

Key Performance Metrics

Response Time

Time taken for the system to respond to a user request

Throughput

Number of transactions processed per unit of time

Concurrent Users

Number of users accessing the system simultaneously

Resource Utilization

CPU, memory, disk I/O, and network usage

Types of Performance Testing

Load Testing

Testing application behavior under expected user loads to identify performance bottlenecks.

Purpose

Measure performance under expected load conditions

Metrics

Response time, throughput, resource utilization

Stress Testing

Testing application behavior beyond normal operational capacity to determine breaking point.

Purpose

Determine system's breaking point and recovery

Metrics

System stability, error handling, recovery time

Scalability Testing

Testing application's ability to scale up or down based on user load and resource allocation.

Purpose

Determine system's capacity to grow

Metrics

Performance at different capacity levels

Endurance Testing

Testing application behavior under sustained load over an extended period to identify memory leaks or degradation.

Purpose

Identify performance degradation over time

Metrics

Memory usage, response time consistency

Priority and Severity in Defect Management

At the time of logging a defect, we must decide the priority and severity of it. Priority and Severity are key attributes in defect management that help teams determine the order and impact of resolving defects.

What is Severity?

  • Definition: Severity defines the impact of a defect on the functionality of the application.
  • Categories: Critical, High, Medium, Low.
  • Focus: It is about the technical impact and is usually assigned by the QA/Test team.

What is Priority?

  • Definition: Priority indicates the urgency with which the defect should be fixed.
  • Categories: High, Medium, Low.
  • Focus: It is about business impact and is usually assigned by the project manager or product owner.

Combinations of Priority and Severity

Here are the four combinations with examples:

High Severity & High Priority

Description: Critical defect that breaks core functionality, making the application unusable. Needs immediate attention.

Examples:

  • Login button not working on a live e-commerce website.
  • Payment gateway crashes during checkout.

High Severity & Low Priority

Description: Major defect, but it occurs in a less frequently used module or functionality. Can be fixed later.

Examples:

  • A crash happens in the admin panel of a rarely used reporting feature.
  • "Forgot Password" feature is not working in a staging environment.

Low Severity & High Priority

Description: Minor defect, but it affects an important area of the application and needs to be fixed quickly.

Examples:

  • Typo in the company name or important heading on the homepage of a live website.
  • Incorrect logo displayed on the landing page.
  • The "Place Order" button on the checkout page is misspelled as "Plce Order."
  • A product page in an e-commerce app displays a price of ₹0 for a popular item due to a data input error.

Low Severity & Low Priority

Description: Minor defect in a less critical module or functionality. Can be fixed later without significant impact.

Examples:

  • Misalignment of text on a non-critical page.
  • Background color inconsistency in a rarely accessed settings page.

Defect Tracking and Reporting Process

Defect tracking and reporting is a crucial part of the software testing lifecycle, ensuring that issues are identified, logged, managed, and resolved systematically.

Step 1: Identify and Log the Defect

When QA execute test cases and find a mismatch between the actual and expected results, we log the issue in a Defect Tracking Tool (e.g., Jira, Bugzilla).

Defect details include:

  1. Summary
  2. Steps to reproduce
  3. Test steps
  4. Test data
  5. Actual result
  6. Expected result
  7. Screenshot/Attachment
  8. Priority
  9. Module tag
  10. Environment
  11. Assigned developer

The initial status of the defect is set to New.

Step 2: Assigning the Defect

Once logged, the defect is assigned to the respective developer or team for investigation.

The status changes to Open or Assigned.

Step 3: Developer Investigation

The developer reviews the defect and there are three possibilities:

  • Valid Defect: They start working on fixing it, and the status is updated to In Progress.
  • Invalid Defect: If the defect is not valid (e.g., works as designed or cannot be reproduced), they may reject it, and the status is changed to Rejected or Not a Bug.
  • Deferred Defect: If the defect is deferred for a future release due to low priority or other reasons, the status changes to Deferred.

Step 4: Fixing the Defect

The developer fixes the defect and updates the status to Fixed or Resolved.

The fix is deployed to the testing environment for verification.

Step 5: Retesting by QA

The QA team retests the defect and there are two possibilities:

  • Fixed Successfully: If the defect is successfully fixed, the status is updated to Closed.
  • Not Fixed: If the defect is not resolved or the issue persists, the status is updated to Reopened, and it is assigned back to the developer.

Step 6: Regression Testing

After closing the defect, QA performs regression testing to ensure the fix hasn't impacted other functionalities.

Note: This step is especially important for high severity defects.

Defect Status Lifecycle

Here's a summarized flow of defect statuses throughout the defect lifecycle:

Status Description Action By

Assigned When the defect is assigned to a developer Test Lead/Manager
In Progress When the developer is actively working on the defect Developer
Deferred When the defect is postponed for later releases Developer/Manager
Rejected/Not a Bug When the defect is invalid or not reproducible Developer
Fixed/Resolved When the developer has implemented the fix Developer
Reopened If the defect is not resolved during retesting QA Engineer
Closed When the defect is verified as fixed and no further action is required QA Engineer

Testing Methodology

Testing Methodology refers to the strategies and approaches used to ensure that software or a system meets the defined quality standards and requirements. It defines how testing is conducted, the objectives, and the methods used.

Black Box Testing

Focus: Functional testing without knowledge of internal code.

Key Techniques:

  • Equivalence Partitioning: Dividing input data into valid and invalid partitions.
  • Boundary Value Analysis: Testing edge cases around boundaries.
  • Decision Table Testing: Evaluating combinations of inputs and expected outcomes.

Examples: User interface testing, functional testing.

White Box Testing

Focus: Testing internal logic and structure of code.

Key Techniques:

  • Statement Coverage: Ensuring every line of code is executed.
  • Branch Coverage: Testing all branches in control structures.
  • Path Coverage: Covering all possible execution paths.

Examples: Unit tests, code reviews.

Gray Box Testing

Focus: Combination of Black Box and White Box techniques.

Purpose: Testing with partial knowledge of the system.

Use Case: Security testing or integration testing.

Static Testing

Focus: Testing without executing the code.

Activities:

  • Code reviews
  • Walkthroughs
  • Inspections
  • Static analysis tools

Purpose: Catch defects early in the development lifecycle.

Dynamic Testing

Focus: Testing by executing the application.

Types:

  • Functional Testing: Validating functionality against requirements.
  • Non-functional Testing: Validating performance, usability, and security.

Manual Testing

Focus: Testing without automation tools.

Types:

  • Exploratory Testing
  • Ad-hoc Testing
  • Regression Testing

Automated Testing

Focus: Using tools and scripts for repeated test execution.

Tools: Selenium, JUnit, TestNG, Pytest, REST Assured.

Advantages: Speed, accuracy, and efficiency for regression testing and large-scale projects.

Agile Testing

Focus: Continuous testing during iterative development.

Principles:

  • Collaboration between developers, testers, and business stakeholders.
  • Test-driven development (TDD) or Behavior-driven development (BDD).

Test-Driven Development (TDD)

Focus: Writing tests before code.

Process:

  1. Write a failing test.
  2. Develop code to pass the test.
  3. Refactor the code and ensure tests still pass.

Behavior-Driven Development (BDD)

Focus: Testing based on expected user behaviors.

Tools: Cucumber, SpecFlow.

Purpose: Align business goals with technical implementation.

Acceptance Testing

Focus: Validating the system meets business needs.

Types:

  • Alpha Testing: Performed by internal testers or users.
  • Beta Testing: Performed by external users in a real environment.

Performance Testing

Focus: Testing system performance under different loads.

Types:

  • Load Testing
  • Stress Testing
  • Scalability Testing
  • Soak Testing

Security Testing

Focus: Ensuring the application is secure from vulnerabilities.

Activities:

  • Penetration Testing
  • Vulnerability Scanning
  • Security Audits

Regression Testing

Focus: Ensuring new code changes do not break existing functionality.

Typically performed after bug fixes, enhancements, or new features are added to the system.

Exploratory Testing

Focus: Testing without predefined test cases.

Goal: Discover defects through creative exploration.

Relies on tester's experience, intuition, and knowledge of the system to find defects that scripted testing might miss.

Testing Terminology

Testing terminology refers to the different approaches used for testing software applications, especially when formal test cases are not predefined. These methods help identify defects when documentation is limited.

Monkey Testing 🐵 (Speed Testing)

Definition:

  • A testing approach used when there are many test cases to execute but limited time.
  • Focuses on executing high-priority test cases first. If time permits, medium and low-priority test cases are executed.

Purpose:

  • Ensures critical functionality is tested quickly.
  • Helps when test execution time is limited.

Example:

A tester quickly validates login, checkout, and payment features in an e-commerce app before a release deadline.

Exploratory Testing 🔍

Definition:

  • A testing approach where testers explore the application without predefined test cases.
  • Selected when testers have domain knowledge and experience.

Purpose:

  • Helps find hidden defects that scripted tests may miss.
  • Encourages creative and critical thinking.

Example:

A tester navigates through a banking app without a test plan, trying different transactions to check for unexpected errors.

Ad-hoc Testing 🎯

Definition:

  • A quick, informal testing approach performed without documentation or planning.
  • Selected when the tester is already familiar with the application.

Purpose:

  • Helps find quick bugs in a short time.

Example:

A tester randomly checks different features of a social media app before release to ensure there are no major issues.

When to Use Each Type?

Monkey Testing

When time is limited, and only high-priority test cases need to be executed.

Exploratory Testing

When testers have domain experience and want to explore functionality dynamically.

Ad-hoc Testing

When testers already know the application and need to perform quick, informal validation.

Difference between Error, Defect, Bugs, Issues

Error

Mistakes found in program by programmers

Defect

Mistakes found by testers

Bugs

Actual defects accepted by developers

Issues

Difficulties in solving bugs by developers

Relationship Analysis

1. If every error can be defect?
No, not all errors become defects
2. If every error can be a bug?
No, only some errors become bugs
3. If every error can be issues?
No, not necessarily
4. If every defect can be a error?
No, defects are found by testers, errors by programmers
5. If every defect can be bug?
No, only when developers accept it
6. If every defect can be issues?
No, only if it's difficult to solve
7. If every bug can be a error?
Yes, bugs originate from errors
8. If every bug can be a defect?
Yes, bugs are accepted defects
9. If every bug can be issue?
No, only if it's difficult to solve
10. If every issue can be error?
No, issues are problem resolutions
11. If every issue can be a defect?
No, issues are broader than defects
12. If every issue can be a bug?
No, issues can be unrelated to bugs

Key Summary

Errors are mistakes made by programmers during development. When testers find these mistakes, they become defects. When developers accept these defects, they become bugs. Issues arise when developers face difficulties in solving these bugs.

Not all errors become defects, and not all defects become bugs. Similarly, not all bugs become issues. The relationship between these terms is hierarchical but not always sequential.

Manual Testing Framework and STLC

A structured manual testing framework ensures comprehensive test coverage, efficiency, and consistency in the testing process. Below are the key steps involved in the Software Testing Life Cycle (STLC):

1

Define Objectives & Scope and Requirement Analysis

  • Understand business requirements and the functional scope and out scope of the application
  • Identify stakeholders and their expectations
  • Document requirements with clear acceptance criteria
  • Identify potential risks and constraints
2

Establish Test Strategy & Planning

Test Strategy (High-Level Approach)

  • Define test levels (Unit, Integration, System, UAT)
  • Identify test methodologies (Functional, Non-functional, Regression, Exploratory)
  • Outline the automation strategy (if applicable)
  • Perform risk analysis and define coverage criteria
  • Define entry and exit criteria for testing phases

Test Planning (Execution-Level Details)

  • Allocate resources and responsibilities
  • Identify test environments, tools, and infrastructure
  • Establish a defect management process (e.g., logging, tracking, triaging defects)
  • Define the test execution schedule and deliverables documentation
  • Estimate testing effort and create timeline
3

Design Test Scenarios & Test Cases

  • Identify test scenarios from requirements and user stories
  • Write detailed test cases covering:
    • Positive & negative scenarios
    • Preconditions, test steps, expected results, and postconditions
    • Test data requirements
    • Priority and severity classifications
  • Implement a traceability matrix to map test cases with requirements
  • Review test cases with stakeholders for completeness
  • Create test data and prepare test environment
4

Execute & Track Testing

  • Perform test execution and log results
  • Conduct different types of testing:
    • Sanity Testing (basic verification)
    • Smoke Testing (critical functionality check)
    • Regression Testing (ensure new changes don't break existing functionality)
    • Ad-hoc and Exploratory Testing
  • Track defects using tools like JIRA, Bugzilla, or TestRail
  • Generate test execution reports for review and analysis
  • Conduct defect triage meetings with stakeholders
  • Perform retesting and regression testing after fixes
5

Review & Continuous Improvement

  • Conduct peer reviews for test cases
  • Identify gaps and areas for improvement, for that we conduct the retrospective meeting
  • Update test artifacts based on requirement changes
  • Maintain a knowledge repository for reusable test cases and best practices
  • Measure testing effectiveness through metrics (defect density, test coverage, etc.)
  • Document lessons learned and process improvements

Benefits of Following STLC

Improved Quality

Systematic approach leads to higher quality software

Cost Efficiency

Early defect detection reduces cost of fixes

Time Savings

Structured process minimizes rework and delays

Better Coverage

Comprehensive test planning ensures all requirements are tested

STLC Best Practices

  • Involve testing team early in the development lifecycle
  • Maintain clear documentation throughout the process
  • Implement requirement traceability to ensure test coverage
  • Regularly update test cases to reflect changing requirements
  • Leverage automation where appropriate to increase efficiency
  • Continuously monitor and improve testing processes
