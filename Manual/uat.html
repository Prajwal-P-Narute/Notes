What is User Acceptance Testing?

User Acceptance Testing (UAT) is the final phase of software testing where real users validate whether the application meets business requirements before production deployment.

Key Points

  • Conducted by business users, stakeholders, or end-users
  • Focuses on business workflows and real-world usage
  • Ensures the software is ready for production
  • Performed in a UAT or pre-production environment
  • Sign-off from stakeholders is required before release

UAT Process

1

Requirements Analysis

Understand business requirements and create test scenarios

2

Test Planning

Define scope, objectives, and approach for UAT

3

Test Case Creation

Develop test cases based on real-world scenarios

4

Test Execution

End-users execute test cases in UAT environment

5

Feedback & Sign-off

Collect feedback and obtain formal approval for release

Types of User Acceptance Testing

Alpha Testing

Alpha testing is conducted in a controlled environment, where testing is done in front of testers, developers, and customers.

Environment: Controlled environment (developer's site)

Participants: Developers, testers, and internal stakeholders

Focus: Identifying bugs before releasing to real users

Beta Testing

Beta testing is conducted in an uncontrolled environment with minimal involvement of developers and testers. Customers test the functionality from their own perspective.

Environment: Uncontrolled environment (user's site)

Participants: Real users and customers

Focus: Gathering feedback on user experience and satisfaction

Alpha vs Beta Testing Comparison

Aspect Alpha Testing Beta Testing
Environment Controlled (developer's site) Uncontrolled (user's site)
Participants Internal testers, developers Real users, customers
Testing Focus Functionality, bugs, and issues User experience, satisfaction
Duration Several weeks Several weeks to months
Feedback Technical issues and bugs User feedback and suggestions
Cost Higher (requires dedicated resources) Lower (users test in their environment)

Re-Testing

What is Re-Testing?

Re-testing is the process of testing a specific fixed defect or bug from a developer to ensure that the issue is resolved.

Key Points of Re-Testing

Purpose: To confirm that the defect has been fixed.

When? Before logging the defect and after a bug fix or code change.

How? Re-execute the failed test cases.

Scope: Focuses only on the specific defect and its related functionality.

Best Practices for Re-Testing

  • Always validate the defect before logging to avoid invalid or duplicate bug reports.
  • Test with multiple test data to ensure the defect is fully resolved and not limited to specific cases.
  • Helps prevent confusion between developers and testers by confirming the validity of the defect before escalation.

Example Scenario

  1. Bug Identification: A tester finds a bug in the login functionality, where entering the correct credentials does not log in the user.
  2. Bug Reporting: The bug is reported to the developer, who fixes the issue.
  3. Re-Testing: The tester retests the same login test case to ensure it works correctly after the fix.

Regression Testing

What is Regression Testing?

In Regression testing, we must check that newly added or newly corrected functionality is working properly, and there is no negative impact on other working modules.

The primary goal of regression testing is to ensure that new changes and fixed bugs do not break the existing functionality.

About Regression Testing

Purpose: Ensure that new changes don't break existing functionality

When to Perform: After major code changes, new feature additions, or bug fixes

Scope: All impacted areas of the application

Test Cases: All relevant test cases, not just the failed ones

Retesting vs. Regression Testing

Feature Retesting Regression Testing
Objective Verify the fixed bug Ensure no new bugs due to changes
Test Cases Only failed test cases All impacted test cases
Execution Performed multiple times if needed Performed after major code changes
Focus Specific defect Overall system stability

Reviews and Types of Reviews

Reviews are a systematic examination of software work products by the author's peers to identify defects and improvements. They are a crucial part of the software development process, helping to improve quality and reduce defects early in the lifecycle.

Types of Reviews

Code Review

Systematic examination of source code to find and fix errors overlooked in the initial development phase.

  • Improves code quality
  • Shares knowledge among team members
  • Finds vulnerabilities and bugs early

Document Review

Evaluation of documentation to ensure accuracy, completeness, and quality.

  • Requirements documents
  • Design specifications
  • User manuals
  • Test plans and cases

Walkthrough

A meeting where the author of the work product explains the product to peers for feedback.

  • Informal review process
  • Led by the author
  • Focus on understanding and education
  • Minimal preparation required

Inspection

Formal, rigorous review process led by a trained moderator to identify defects.

  • Formal process with defined roles
  • Uses checklists
  • Formal follow-up process
  • Metrics are collected

Technical Review

Peer discussion focused on achieving consensus on the technical approach.

  • Evaluates technical concepts
  • Focuses on alternatives and correctness
  • Performed by technical experts
  • Less formal than inspections

Peer Review

Evaluation of work product by colleagues of similar competence.

  • Improves quality through collective ownership
  • Promotes knowledge sharing
  • Helps in identifying defects early

Benefits of Reviews

  • Early Defect Detection: Find and fix issues early in the development process
  • Cost Reduction: Reduce cost of fixing defects found later in the lifecycle
  • Knowledge Transfer: Share knowledge and best practices among team members
  • Quality Improvement: Consistently improve the quality of software products

Review Process

  1. Planning: Define scope, objectives, and select participants
  2. Preparation: Review materials are distributed and studied by participants
  3. Examination: Review meeting where defects are identified and discussed
  4. Rework: Author addresses the identified issues
  5. Follow-up: Moderator verifies that all issues have been addressed
