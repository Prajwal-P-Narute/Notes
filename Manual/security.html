What is Security Testing?

Security Testing is the process of finding and fixing weaknesses in a system or application to ensure it is safe from hackers and unauthorized access.

Purpose of Security Testing

To ensure the application is secure from threats like hacking, data theft, or unauthorized access.

Security testing checks:

  • If only authorized people can access the system
  • If sensitive data is safe
  • If the system can handle attacks without breaking

Security Testing Techniques

Vulnerability Scanning: Checking for weak spots in the system

Penetration Testing: Simulating real-world attacks to find risks

Security Auditing: Reviewing logs and code for security issues

Authentication vs Authorization

1 Verifies if the user's identity is valid or invalid to grant access Verifies if the user has the necessary permissions to access specific resources
2 How it works:
- Username and password: Basic login method
- Biometrics: Fingerprint, face recognition
- Two-factor authentication (2FA): Password + one-time password (OTP) 		How it works:
Based on roles (e.g., admin, user) and permissions (e.g., read, write)

Encryption vs Decryption

Sr no Encryption Decryption
1 Changes readable data into an unreadable format to keep it safe from unauthorized access Converts encrypted data back into original readable data for authorized users
2 Protects sensitive data like passwords, credit card info, or personal details when it's stored or transmitted Allows only authorized users to access the original data after encryption
3 Common methods:
- Symmetric encryption (same key to encrypt and decrypt)
- Asymmetric encryption (public key to encrypt, private key to decrypt) 		Common methods:
- AES for symmetric encryption
- RSA for asymmetric encryption

Security Testing Types

Penetration Testing

Purpose: Simulates real cyberattacks to find weaknesses

Example Scenario: Hacking into a system using SQL Injection to test security

Vulnerability Scanning

Purpose: Identifies security loopholes automatically

Example Scenario: Detecting outdated software with known security flaws

Authentication & Authorization Testing

Purpose: Ensures proper user access controls

Example Scenario: Attempting to bypass login or access unauthorized pages

Real-World Examples

Authentication & Authorization

When you log into a website (authentication), and then you can only access certain pages based on your membership level (authorization).

Encryption & Decryption

When you send an email with sensitive information, it's encrypted to protect it while traveling over the internet. The recipient uses a key to decrypt it and read it.

Hot Fix

A Hot Fix is an urgent patch deployed directly to production to resolve a critical issue (like downtime, payment failure, or security flaw) that cannot wait for the next scheduled release.

Key Points

Purpose: Fix critical business-blocking issues immediately.

Priority: Very high, deployed in hours (not days/weeks).

Code Change: Minimal, focused only on the defect.

Process: Bypasses normal release cycle, directly moved to production.

Testing: Smoke testing + focused regression on impacted areas, followed by full regression later.

Outcome: Ensures business continuity and reduces downtime.

Example Scenario

"Suppose the payment gateway stops working in production. This impacts revenue, so the team raises a hot fix. Developers apply a minimal code change, and as QA, I perform quick impact analysis, run smoke tests on payment and checkout flows, validate the fix in production, and later ensure it is covered in the next regression cycle."

Hot Fix Process Flow:

  1. Critical Issue Identified: Payment gateway failure in production
  2. Hot Fix Initiated: Bypasses normal release cycle
  3. Minimal Code Change: Focused only on the defect
  4. Targeted Testing: Smoke + focused regression testing
  5. Production Deployment: Fix validated in live environment
  6. Full Regression Later: Included in next regular release cycle

Quick Summary

"A hot fix is an urgent production patch to resolve critical issues like downtime or payment failure. It bypasses the normal release cycle, involves minimal code changes, and requires immediate testing—smoke and focused regression—followed by validation in production and later full regression."

When to Use a Hot Fix:

  • Critical production issues affecting business operations
  • Security vulnerabilities that need immediate attention
  • System downtime or major functionality failure
  • Issues that cannot wait for the next scheduled release
