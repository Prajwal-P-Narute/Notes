What is Black Box Testing?

Black box testing, also known as system integration testing, functional testing, or detail level testing, is a software testing method where the internal structure/design/implementation of the item being tested is not known to the tester. The tester only focuses on the input and output of the software system without knowing how the system processes the input to produce the output.

Key Characteristics

Tester Involvement: Testers are primarily involved in this testing approach, focusing on the external behavior of the system.

Testing Scope: Involves checking the overall functionality including both internal and external aspects of the application.

System & Function Testing: Includes functionality testing (high involvement), usability testing (high involvement), security testing (less involvement), and performance testing (less involvement).

Functionality Testing

Functionality testing is the process of checking correctness and completeness of functionality. It ensures that the software application functions according to the specified requirements.

Functional Testing

Focuses on the internal functionality of the application including:

  • Behavioral coverage
  • Input domain coverage
  • Error handling coverage
  • Back-end coverage
  • Service level coverage
  • Calculation base coverage

Non-functional Testing

Focuses on the external functionality and quality attributes of the application including:

  • Recovery testing
  • Compatibility testing
  • Configuration testing
  • Inter system testing
  • Installation testing
  • Globalization testing
  • Gold Plating
  • Parallel testing

Functional Testing Coverage Details

1. Behavioral Coverage

In behavioral coverage, testers check the property and behavior of objects within the application.

Example: In a bus booking application, when a user hovers over a seat radio button, it gets highlighted with a bright color (behavior). After clicking, the seat gets booked (property).

2. Input Domain Coverage

This coverage focuses on validating the size and data type of inputs using techniques like Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP).

Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP)

Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP) are two widely used techniques in software testing to identify test cases efficiently.

Equivalence Class Partitioning (ECP)

What it does: It divides input data into groups or "partitions" where all inputs in the same group are expected to behave similarly.

Why use it: To reduce the total number of test cases while still covering maximum functionality.

How it works:

  1. Identify all possible inputs for a feature or function
  2. Group these inputs into "valid" and "invalid" partitions
  3. Pick one representative value from each partition to test

Example: For a login form requiring a password of 8 to 12 characters:

  • Valid Partition: Inputs with 8 to 12 characters (e.g., "password123")
  • Invalid Partitions:
    • Less than 8 characters (e.g., "pass")
    • More than 12 characters (e.g., "passwordislonger")

Boundary Value Analysis (BVA)

What it does: Focuses on testing values at the "edges" of input ranges because errors are more likely there.

Why use it: To catch boundary-related defects.

How it works:

  1. Identify the boundaries of input ranges (minimum, maximum, and just outside these limits)
  2. Test values on, just below, and just above these boundaries

Example: Using the same password example (8 to 12 characters):

  • Boundaries:
    • Minimum: 8 characters
    • Maximum: 12 characters
  • Test cases:
    • On the boundary: 8, 12 characters (e.g., "password", "pass123456")
    • Just outside: 7, 13 characters (e.g., "pass123", "passwordplus")

3. Error Handling Coverage

This coverage ensures the system displays appropriate error messages when invalid inputs or actions occur.

Example: When a user enters only 2 digits in a 10-digit mobile number field, the system should display an error message like "Please enter a 10-digit mobile number" and highlight the field in red.

4. Back-end Coverage

This coverage verifies that data is correctly stored in and retrieved from the database.

Example: In a financial app, all personal information provided during account opening (name, mobile number, PAN, etc.) should be correctly stored in the database.

5. Service Level Coverage

This coverage ensures the system works according to the function flow diagram created by business analysts.

Example: In a cloud storage app, the sequence of pages (home page → folders → saved data) should follow the predefined flow without deviations.

6. Calculation Base Coverage

This coverage validates that arithmetic operations within the application work correctly.

Example: In an e-commerce app, purchasing 2 items priced at ₹1500 each should correctly calculate the total as ₹3000. Adding 2 more items should update the total to ₹6000.

Non-functional Testing Types

1. Recovery Testing / Reliability Testing

Process of checking whether an application can recover from abnormal situations to normal operation.

Example: A result website that becomes unresponsive under heavy load should return to normal operation when the load decreases.

2. Compatibility Testing

Ensures that a software application works correctly across different environments, including browsers, operating systems, devices, networks, and hardware configurations.

Forward vs. Backward Compatibility
Criteria Forward Compatibility Backward Compatibility
Definition Tests software on newer versions Tests software on older versions
Purpose Ensures app works with future updates Ensures app works with legacy systems
Example Checking an Android 12 app on Android 14 Checking a Word 2024 file in Word 2016

3. Configuration Testing

Focuses on how the software performs with different system configurations (hardware, OS, network settings).

Example: Testing an application on Windows with different RAM configurations (4GB, 8GB, 16GB) to ensure proper functionality across all setups.

4. Inter System Testing

Validates the interaction and data flow between multiple integrated systems to ensure they communicate correctly.

Example: Testing how a CRM system integrates with an email marketing platform to ensure customer data syncs correctly between both systems.

5. Installation Testing

Verifies if the application installs correctly on the target platform and uninstalls without leaving residual files.

Key Checks:

  • Application creates desktop shortcuts after installation
  • All required files, packages, and drivers are present
  • User interface is intuitive and easy to navigate
  • Sufficient disk space validation before installation
  • Proper uninstallation without residual files

6. Globalization Testing

Verifies whether an application supports different languages, regions, and cultural conventions without requiring code changes.

Example: A news app should correctly display content in different languages, maintain proper formatting for numbers, dates, and currency symbols, and support right-to-left text for languages like Arabic.

Key Aspects:

  • Text translation while keeping numbers in English format
  • Correct date, time, currency, and number formats for different regions
  • Text expansion handling when translated to different languages
  • UI adaptability including alignment, font support, and RTL text handling

7. Gold Plating

Testing of extra features not mentioned in customer requirements.

Important: If developers add extra features with customer permission (and potential additional payment), testers do not log defects. If added without permission, testers should log these as defects.

Benefits of Black Box Testing

Independent Testing: Testers don't need knowledge of internal code, preventing developer bias

User Perspective: Tests from the end-user's viewpoint, ensuring real-world usability

Early Detection: Helps identify specification gaps and ambiguities early in the cycle

Efficient for Changes: Code changes don't affect test cases, making maintenance easier

Integration Testing

Integration is the process of mapping new modules with existing modules. Integration Testing is the process of verifying the interaction and data flow between interconnected modules/components of an application.

Main Purpose of Integration Testing

Focus on validating data flow, error handling, and module interactions.

Data Flow Validation: Ensure data passes correctly between modules

Error Handling: Verify proper error handling across module boundaries

Module Interactions: Test interfaces and interactions between modules

Types of Integration Testing

Big Bang Approach

All modules are integrated and tested together.

Pros: Simple, Suitable for small systems

Cons: Difficult to catch defects, Debugging is challenging

Incremental Approach

Modules are integrated and tested step by step.

Top-Down: Starts with high-level modules and moves to lower-level ones

Bottom-Up: Starts with lower-level modules and progresses upward

Hybrid/Sandwich: Combines both top-down and bottom-up approaches

Stub and Driver-Based

Used when modules are dependent but incomplete.

Stub: A dummy implementation for lower-level module that provides responses to the higher-level module

Driver: A dummy implementation for higher-level module that supplies inputs to the lower-level module

Tools Used in Integration Testing

  • Postman / SoapUI: For API-level testing during integration
  • JUnit / TestNG: For testing Java-based integrations
  • Selenium: For UI-level integration testing
  • REST Assured: For API integration testing in Java

Integration Testing Process

  1. Identify Modules: Determine which modules need to be integrated
  2. Plan Integration: Decide on integration approach and strategy
  3. Create Test Cases: Develop test cases focusing on interfaces
  4. Execute Tests: Run integration tests and monitor interactions
  5. Report & Fix: Document issues and work with developers to fix them

Example Scenario

1. E-commerce Application: An online store has separate modules for user authentication, product catalog, shopping cart, and payment processing

2. Integration Testing: Testing how the shopping cart module interacts with the product catalog and payment processing modules

3. Test Cases: Verify that adding a product to cart updates the total correctly, and that checkout process communicates properly with payment gateway

4. Issues Found: Payment module doesn't handle failed transactions correctly, causing cart to not reset properly

5. Resolution: Developers fix the error handling logic and integration testing verifies the fix
