What is SDLC?

Software development life cycle consists of set of particular stages which start from information gathering to the maintenance of product.

Definition

The Software Development Life Cycle (SDLC) is a structured process that enables the production of high-quality, low-cost software, in the shortest possible production time. The goal of the SDLC is to produce superior software that meets and exceeds all customer expectations and demands.

Stages in SDLC

1Information Gathering

BA is responsible for gathering the information from customers. Information gathering is nothing but collecting requirement from customers. Information gathering involves business requirements specification (BRS). BRS is bridge between client to developers & testers. BRS is a document that contains all (overall) requirements related to the application. BA is responsible for preparing BRS documents.

2Analysis

Here also BA is responsible person. In this phase BA will prepare software requirement specification (SRS). SRS is detail level documents. SRS document is also known as functional requirement specification.

SRS Components:

  • Functional flow diagram
  • Functional requirements
  • Use cases
  • Snapshots

3Design

High Level Design: Contains the working of main module. It includes relation and dependency of main module. It includes what and how any module works. High level design is created by design Architecture.

Low Level Design: Contains the working of sub modules. Low level design is created by Front end developer. Ex. If any sign-up page is main module, then name, password, mobile no etc. are the sub functionality of main module.

4Coding

It means developing the codes. One line is code. Multiple line of code is program. Set of programs written by developer creates the software.

Code is written by three types of developer:

  • Front end developer: UI functionality, function flows, process are developed by the front-end developer
  • Back-end developer: Data management, Data gathering, Data security, Algorithm selection is done by back-end developer
  • Full stack developer: Person/developer who can work as front end as well as back-end developer known as full stack developer

5Testing

Testing is the process checking a completeness and correctness of the software.

White Box Testing Black Box Testing
It is also called as Unit code level testing, and clear box testing Black box testing is also known system and function testing
Developer is responsible for white box testing Tester is responsible for black box testing
Developer will debug or compile the code and check the code if any error is there Tester has to check overall functionality (internal and external) of the application
Developer will test only positive scenario Tester has to test app in positive as well as negative way
Aim of white box testing completeness and correctness of code and program Positive testing: providing correct input to the application
Negative testing: providing negative inputs

Gray Box Testing: It is combination of white and black box testing. To do gray box testing tester should have some programming knowledge.

6Maintenance

Maintenance means provide service after delivery of product/software in free of cost. Maintenance involves non-technical support as well as technical support. Non-technical support called as BPO (Business process outsource). Technical support is called as KPO (Knowledge process outsource).

SDLC Process Models

Fish Model: The Fish Model is an advanced version of the SDLC that consists of verification and validation processes working together to ensure software quality.

Verification Validation
Verification is the process of checking whether the developed product fulfills the customer's requirements or not Validation is the process of checking whether the developed product is free from bugs or not
Verification is done in different stages of SDLC Validation is done after complete development and at the testing stage
It is also known as static testing It is also known as Quality control or dynamic testing
The focus of verification is on methods/process like reviews, inspection and walk through etc. The focus of validation is on Quality of software, so we can deliver software free from bugs
It helps to find the bug in early stage It helps to increase the quality of software
Quality assurance team does the verification Quality analyst team does the validation
Budget required less for verification as compared to validation Budget required for validation is more than verification

Waterfall Model: Waterfall model is step by step implementation of SDLC. In waterfall model requirements are fixed. If first stage is complete and process goes to next phase, then we cannot revert back to the first phase. Generally, waterfall model is used in small scale company or small application. The general time duration of waterfall model is 3 months. In waterfall model, when tester finds any bug then they do not send that back to the developer, instead they log the bug, make report & those bugs get fixed in next version of software.

V-Model: V stands for verification and validation. V model has working in which verification and validation phases run parallel. In V model development stages are mapped with testing stages. In V model, if one stage gets completed and 2nd is running but changes occurred in 1st phase then we can revert back to 1st phase but customer have to pay some amount to company. The duration of V model is 3 months. It is used for large application. Generally, this method is not used nowadays.
