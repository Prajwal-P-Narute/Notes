Expenses Manager Requirements

Complete overview of all sprint requirements with acceptance criteria

Sprint 1 Sprint 2 Sprint 3 Sprint 4 Sprint 5

Sprint 1 Requirements

Default Reimbursable Option for Debit Transactions

When adding a debit transaction on the Add Transaction page, the "No" option for Reimbursable should be selected by default.

Acceptance Criteria
  • The Reimbursable field should default to "No" when transaction type is Debit
  • User can manually change the selection if needed

Sort Transactions by Latest First

On the Expenses Tracker page, transactions should be listed from the most recent to the oldest (descending order by date).

Acceptance Criteria
  • Default view shows latest transactions at the top
  • Sorting logic should be applied automatically upon page load

Pagination (15-Entries)

Implement pagination on the tracker page, grouping transactions in 15 entries for every oage.

Acceptance Criteria
  • Each page should display 15 entries
  • Navigation controls (Previous, Next) should be functional and intuitive

Mobile Responsiveness

Ensure the Expenses Tracker page is fully responsive and optimised for mobile device usage.

Acceptance Criteria
  • Layout should adapt to different screen sizes without horizontal scrolling
  • All features should remain accessible and usable on mobile devices

Prevent Duplicate Entries

Implement functionality to avoid duplicate transaction entries after submission by reloading or resetting the form.

Acceptance Criteria
  • After successfully adding a transaction, the page should either refresh or the form should reset automatically
  • Display a success confirmation message to the user
  • Duplicate data should not be stored in the database

Sprint 2 Requirements

Number Formatting

All numerical values displayed on the UI should follow a consistent format (e.g., comma-separated thousands, two decimal places if required).

Acceptance Criteria
  • Numbers should be human-readable (e.g., 1,000 instead of 1000)
  • No loss of precision for decimal values

Highlight Credit and Debit Text

The labels Credit and Debit on the Expense Tracker page should be displayed in bold for better visibility.

Acceptance Criteria
  • Font weight set to bold
  • No impact on other labels' formatting

Display Total Expense and Total Income

Add two new fields to the Expense Tracker page to display Total Expense and Total Income.

Acceptance Criteria
  • Totals should update dynamically based on entries
  • Formatting should be consistent with number formatting requirements

Increase Table Height in Tracker Page

Adjust the height of the table in the Tracker page to enhance data visibility and reduce scrolling.

Acceptance Criteria
  • Table height should accommodate more rows without breaking page layout
  • Responsive design compatibility must be maintained

Account-wise Tracker

Implement functionality to view and filter transactions based on selected accounts.

Acceptance Criteria
  • User can select an account and view only related transactions
  • Totals should update according to the selected account

Masked Data Storage in Database

Sensitive data should be stored in the database in a masked or encrypted format for security.

Acceptance Criteria
  • Masking/encryption must follow security best practices
  • Retrieval should display only allowed visible portions (e.g., XXXX1234)

Sprint 3 Requirements

Forgot Password Functionality

Enable users to reset their password if they have forgotten it.

Acceptance Criteria
  • User can request a password reset using registered email/phone
  • Secure verification (OTP/email link) for identity confirmation
  • New password stored in encrypted format
  • User can log in with the new password

Opening Balance Input on First Registration

Allow users to set an Opening Balance (OB) during registration.

Acceptance Criteria
  • OB field available during first registration
  • OB is optional
  • Default OB is 0 if left blank

Category-wise Filter

Enable users to filter transactions by category.

Acceptance Criteria
  • User can select category from dropdown/list
  • Transactions filtered based on category
  • Summary of transactions updates accordingly

Date Range Filter

Allow filtering of transactions by date range.

Acceptance Criteria
  • User can select start and end dates
  • Only transactions within date range are shown
  • Summary of transactions updates accordingly

Summary of Transaction Color Coding

Visually differentiate transaction summaries with colors and bold text.

Acceptance Criteria
  • Total: Blue, bold
  • Income: Green, bold
  • Expenses: Red, bold

Persistent Login

Users should remain logged in until they explicitly log out.

Acceptance Criteria
  • Session remains active after browser close/refresh (within token validity)
  • Manual logout required to end session
  • No auto logout unless session/token expiry

Disable Button with Loader

Prevent multiple submissions by disabling button once clicked.

Acceptance Criteria
  • Button disabled immediately on click
  • Loader + 'Please wait' shown until process completes
  • Button re-enabled after completion
  • No duplicate requests triggered

Cloud Server Migration

Migrate application hosting from Render to Koyeb.

Acceptance Criteria
  • Application deployed successfully on Koyeb
  • APIs, DB, and integrations work without errors
  • Performance equal or better than Render
  • No downtime during/after migration

Sprint 5 Requirements

Edit Transaction

As a user, I should be able to edit an existing transaction so that I can correct mistakes or update details such as amount, category, label, or description.

Acceptance Criteria
  • User should be able to select an existing transaction and edit its fields
  • Updated values should reflect immediately in the transaction list
  • Calculations should update automatically based on the new transaction details
  • System should maintain data consistency and not duplicate records
Impact
  • Current Balance, Total Expense, and Total Income will be recalculated after edit
  • Data accuracy will improve by allowing correction of errors

Delete Transaction

As a user, I should be able to delete a transaction so that I can remove unwanted or incorrect entries.

Acceptance Criteria
  • User should be able to select a transaction and delete it
  • Deleted transaction should be removed from the transaction list
  • Calculations should update correctly after deletion
  • System should prevent accidental deletions by confirmation prompt
Impact
  • Current Balance, Total Expense, and Total Income will be recalculated after deletion
  • Database will maintain only valid transactions

Master Module for Category

As a user, I should be able to manage (add, update, delete) expense and income categories so that I can personalize the classification of my transactions.

Acceptance Criteria
  • User can add new categories under Debit or Credit type
  • User can edit the name of an existing category (including default)
  • User can delete any category unless it is currently linked to an existing transaction
  • UI should display updated category list in dropdowns

  • Flexibility for personalization
  • Possible re-mapping needed when deleting default categories

Remove Reimbursable (Y/N) Column from UI

Remove the "Reimbursable (Y/N)" column from the UI since it is not needed, but keep backend functionality intact.

Acceptance Criteria
  • Reimbursable column should no longer be visible on transaction listing screens
  • Backend logic should continue to work
  • Existing reports or calculations dependent on reimbursable should continue working
Impact
  • Cleaner UI
  • No functional change in calculations, only visual removal

Add Label Functionality for Transactions

As a user, I want to add labels to my transactions (like "Essential", "Luxury", "Recurring") so that I can analyze my necessary and unnecessary expenses in more detail.

Acceptance Criteria
  • User should be able to create, update, and delete labels as a master list
  • While adding/editing a transaction, user should be able to assign one or more labels
  • Labels are non-mandatory
  • Labels should be displayed along with the transaction in listing screens
  • User should be able to filter transactions by label
Impact
  • New labels field will be added to transaction table
  • Users gain advanced categorization for analysis
  • Filtering and reporting logic will be extended to include labels
