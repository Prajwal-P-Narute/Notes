Software Testing Manual

Comprehensive guide to software testing concepts, methodologies, and best practices

Introduction to Software Testing

Software is the collection of specialized programs which take user input & generate desired output.

Software testing is the process of verifying that a given software or application is generating the desired output or not. It is the process of checking completeness and correctness of the software with respect to customer expectations.

Resources involved in Software Development

  • Customer - Provides requirements and expectations
  • Business Analyst - Gathers and documents requirements
  • Developer - Develops application as per requirements
  • Tester - Checks correctness & completeness of functionality
  • End User - Ultimately uses the final product

Software Quality Assurance (SQA)

SQA is communication between customer & BA. It ensures the product quality by considering several factors:

Meeting Requirements

Identify type of application (banking, telecom, healthcare, finance, etc.) and ensure it meets all specified requirements.

Meeting Expectations

Ensure privacy and security of the application, and verify performance under heavy load conditions.

Cost Management

Manage project costs as per customer requirements and available resources.

Timely Delivery

Deliver the project within the agreed timeframe to avoid penalties (escalation).

Maintenance

Provide service after delivery and fix any post-delivery issues free of cost.

Key SQA Factors

  • To meet customers' requirements - Identify type of application (banking, telecom, healthcare, finance, etc.)
  • To meet customers' expectations
    • Privacy → includes security of application
    • Performance → application should be stable under heavy load
  • Cost of project → Decided as per customer requirements & resources
  • Timing of delivery → Discussed during requirement gathering. Delay may cause penalty (escalation)
  • Maintenance → Service provided after delivery. Any post-delivery issues must be fixed free of cost

Software Development Life Cycle (SDLC)

Software development life cycle consists of set of particular stages which start from information gathering to the maintenance of product.

Stages in SDLC

  1. Information gathering
  2. Analysis
  3. Design
  4. Coding (developer)
  5. Testing
  6. Maintenance

1. Information Gathering

  • BA is responsible for gathering the information from customers
  • Information gathering is nothing but collecting requirement from customers
  • Information gathering involves business requirements specification (BRS)
  • BRS is bridge between client to developers & testers
  • BRS is a document that contains all (overall) requirements related to the application
  • BA is responsible for preparing BRS documents

2. Analysis

  • Here also BA is responsible person
  • In this phase BA will prepare software requirement specification (SRS)
  • SRS is detail level documents
  • SRS document is also known as functional requirement specification

SRS Components:

  • Functional flow diagram
    • It is flow of our application
    • This flow shows relation between the tasks
    • It will give the idea about sequence of functionality
    • Relationship of functionality is nothing but dependency of each other
    • Ex (sign up page → login page → home page → account page)
  • Functional requirements
    • It is nothing but attributes which are required to complete a specific function
    • Ex. Sign up page - for sign up page required 1st name, last name, mob no, email id, passwords, sign up button
    • Ex. For name should be some requirements
      1. Name should be in character only
      2. Name should not accept number
      3. It should not accept any blank space
      4. It should not accept any special symbols
  • Use cases
    • Use cases is nothing but functionality in term of input and output
    • Use cases is nothing but users how to deal with functionality to achieve target
    • It help to create foundation for creating user stories or test cases in agile methodologies
  • Snapshots
    • Snapshots are visualization of any functionality before development of any application
    • Snapshot are created by BA
    • Snapshot give idea to developer and designer that how software supposed to look like

Software Development Process Models

Fish Model

The Fish Model is an advanced version of the SDLC that consists of verification and validation processes working together to ensure software quality.

Verification vs Validation

Verification Validation
Verification is the process of checking whether the developed product fulfills the customer's requirements or not Validation is the process of checking whether the developed product is free from bugs or not
Verification is done in different stages of SDLC Validation is done after complete development and at the testing stage
It is also known as static testing It is also known as Quality control or dynamic testing
The focus of verification is on methods/process like reviews, inspection and walk through etc. The focus of validation is on Quality of software, so we can deliver software free from bugs
It helps to find the bug in early stage It helps to increase the quality of software
Quality assurance team does the verification Quality analyst team does the validation
Budget required less for verification as compared to validation Budget required for validation is more than verification

Waterfall Model

  • Waterfall model is step by step implementation of SDLC
  • In waterfall model requirements are fixed
  • If first stage is complete and process goes to next phase, then we cannot revert back to the first phase
  • Generally, waterfall model is used in small scale company or small application
  • The general time duration of waterfall model is 3 months
  • In waterfall model, when tester finds any bug then they do not send that back to the developer, instead they log the bug, make report & those bugs get fixed in next version of software

V-Model

  • V stands for verification and validation
  • V model has working in which verification and validation phases run parallel
  • In V model development stages are mapped with testing stages
  • In V model, if one stage gets completed and 2nd is running but changes occurred in 1st phase then we can revert back to 1st phase but customer have to pay some amount to company
  • The duration of V model is 3 months
  • It is used for large application. Generally, this method is not used nowadays

Agile Methodology

Agile Methodology is a way of working in software development that focuses on delivering small parts of a project step-by-step instead of completing everything at once. The idea behind this is to be flexible, work closely with the team and customers, and make changes as needed.

Key Points:

  • Work in Sprints: Projects are divided into smaller pieces (called sprints) so teams can deliver updates regularly
  • Customer Involvement: Customers give feedback, so the product meets their expectations
  • Quick Changes: Agile allows teams to make changes easily if requirements change
  • Team Collaboration: Everyone works together and shares ideas to solve problems
  • Continuous Improvement: Teams review their work for continuous improvements

Popular Frameworks in Agile:

  • Scrum: Teams work in sprints, and there are meetings like daily standups to track progress
  • Kanban: Tasks are managed visually using boards to track progress

Benefits:

  • Faster delivery of features
  • Happier customers due to regular updates
  • Better teamwork and communication
  • Fewer mistakes because of frequent testing

Software Testing Levels

Software testing is typically performed at different levels to ensure comprehensive coverage and early defect detection.

Unit Testing

Testing individual components or modules of the software in isolation from the rest of the system.

Key Characteristics:

  • Performed by developers during the coding phase
  • Focuses on internal program logic and code structure
  • Uses white-box testing techniques
  • Helps identify defects early in the development cycle
  • Typically automated using frameworks like JUnit, NUnit, or TestNG

Integration Testing

Testing the interaction between integrated units or components to verify they work together as expected.

Integration Approaches:

Big Bang Integration

All components are integrated simultaneously and tested as a whole.

Top-Down Integration

Testing starts from the top-level modules and progressively incorporates lower-level modules.

Bottom-Up Integration

Testing begins with lower-level modules and progressively adds higher-level modules.

Sandwich/Hybrid Integration

Combination of top-down and bottom-up approaches.

System Testing

Testing the complete, integrated system to verify that it meets specified requirements.

System Testing Includes:

  • Functionality testing
  • Performance testing (load, stress, volume)
  • Security testing
  • Usability testing
  • Compatibility testing
  • Installation testing
  • Recovery testing

Acceptance Testing

Final testing phase performed to determine whether the system is ready for release.

Types of Acceptance Testing:

Alpha Testing

Performed by the internal test team within the organization.

Beta Testing

Performed by a limited number of actual users in their environment.

Contract Acceptance Testing

Performed to verify compliance with contractual requirements.

Regulation Acceptance Testing

Performed to verify compliance with governmental, legal, or safety regulations.

Software Testing Types

Integration Testing

Integration is the process of mapping new modules with existing modules. Integration Testing is the process of verifying the interaction and data flow between interconnected modules/components of an application.

Main Purpose of Integration Testing

Focus on validating data flow, error handling, and module interactions.

Types of Integration Testing

Big Bang Approach

All modules are integrated and tested together.

  • Pros: Simple, suitable for small systems
  • Cons: Difficult to catch defects, debugging is challenging

Incremental Approach

Modules are integrated and tested step by step.

  • Top-Down: Starts with high-level modules and moves to lower-level ones
  • Bottom-Up: Starts with lower-level modules and progresses upward
  • Hybrid/Sandwich: Combines both top-down and bottom-up approaches

Stub and Driver-Based

Used when modules are dependent but incomplete.

  • Stub: A dummy implementation for lower-level module that provides responses to the higher-level module
  • Driver: A dummy implementation for higher-level module that supplies inputs to the lower-level module

Tools Used in Integration Testing

  • Postman or SoapUI: For API-level testing during integration
  • JUnit/TestNG: For testing Java-based integrations
  • Selenium: For UI-level integration testing
  • REST Assured: For API integration testing in Java

Sanity Testing

It is also known as zero level testing or built verification testing. The aim of sanity testing is to check basic or core functionality of the application.

Working

  • Whenever integration test ends, zero level testing starts
  • Database analyst team create a test environment by creating URL which is useful to run the built
  • After creating test environment, tester tests and checks if it generated desired output or not
  • In sanity testing, Tester tests for:
    • Basic core functionality
    • Tab validation
    • Link validation
    • Page validation
    • GUI validation (Graphical User Interface)

Sanity Testing Components

Basic Core Functionality

Testing buttons, icons, forms which allow users to proceed to next stages.

Example: In Facebook application, testing if the submit button on the sign up page works correctly.

Tab Validation

Testing the functionality of input fields and tabs.

Example: In Facebook application, verifying that characters, symbols, numbers entered via keyboard appear correctly in input fields.

Link Validation

Checking the sequence of interlinked pages work correctly.

Example: In Amazon application, testing payment process links work correctly with UPI or third-party applications.

Page Validation

Checking navigation between pages works correctly using next and back buttons.

Example: In WPS application, testing PDF page navigation works properly.

GUI Testing

Validating the visual interface where users interact directly.

Example: Checking if pages display correctly, images are clear, and there's no blurring anywhere on the page.

Smoke Testing

It is an advanced version of sanity testing. Back-end developers and testers are involved in smoke testing.

Smoke test = sanity test + trouble shooting = sanity test + package validation

Package Validation

Package is a collection of objects that is downloaded to our system when we install a build.

Functionality Testing

It is the process of checking correctness and completeness of functionality. There are 2 types of functionality testing:

  1. Functional testing
  2. Non-functional testing

Functional Testing

In functional testing, we must check internal functionality of the application. It includes:

Behavioral Coverage

Checking property and behavior of the object.

Example: In RedBus application, selecting a specific seat highlights it (behavior) and books it (property).

Input Domain Coverage

Checking size and data type of input using BVA and ECP techniques.

Example: Telegram mobile number field accepts exactly 10 digits (BVA) and rejects characters/symbols (ECP).

Error Handling Coverage

Checking that system shows appropriate error messages.

Example: Telegram shows "please enter 10-digit mobile no" when less digits are entered.

Back-end Coverage

Verifying data is correctly stored and retrieved from database.

Example: In Upstox app, verifying all account opening information is saved in database.

Service Level Coverage

Checking system works according to function flow diagram.

Example: Google Drive app follows correct page sequence: home page → folders → saved data.

Calculation Base Coverage

Verifying arithmetic operations work correctly.

Example: Amazon correctly calculates total price for multiple quantities of products.

Non-functional Testing

It is process of checking external functionality of application. Non-functional testing includes:

Recovery Testing

Checking application can recover from abnormal to normal situations.

Example: Result website recovers after too many students access it simultaneously.

Compatibility Testing

Ensures software works across different environments, browsers, OS, devices.

Forward Compatibility Backward Compatibility
Tests software on newer versions Tests software on older versions
Ensures app works with future updates Ensures app works with legacy systems

Configuration Testing

Testing how software performs with different system configurations.

Example: Testing application on Windows with different RAM sizes (4GB, 8GB, 16GB).

Inter System Testing

Validating interaction and data flow between multiple integrated systems.

Installation Testing

Verifying build installs correctly on user expected platform.

  • Application creates desktop shortcut
  • All required files are present
  • User-friendly interface
  • Disk space validation
  • Proper uninstallation

Globalization Testing

Verifying application supports different languages, regions, and cultural conventions.

Example: Daily News Hunt app correctly displays content in multiple languages with proper formatting.

Gold Plating

Testing extra features not mentioned in customer requirements.

If developer adds features without permission, tester logs defects.

Parallel Testing

Running same tests on different versions/system configurations simultaneously.

Usability Testing

Evaluates how easy and user-friendly a system is. Not the same as Accessibility Testing.

Main Purpose

  • Improves customer satisfaction
  • Reduces support costs
  • Identifies UX issues early

Aspects of Usability Testing

GUI Testing

Verifying graphical user interface is user-friendly and aligned with expectations.

Example: E-commerce checkout button size and contrast affecting success rates.

Manual Support Testing

Ensuring users can easily access help resources when needed.

Example: Banking app provides clear guidance when OTP entry fails.

Performance Testing

Checking the speed and processing capability of our build under various conditions.

Types

  • Load Testing: Tests system behavior under expected user loads
  • Stress Testing: Determines system behavior under extreme conditions
  • Spike Testing: Examines how system handles sudden traffic spikes

Re-Testing

Testing specific fixed defects to ensure issues are resolved.

Key Points

  • Purpose: Confirm defects have been fixed
  • When: After bug fixes or code changes
  • How: Re-execute failed test cases
  • Scope: Focuses only on specific defects

Checking that new changes don't negatively impact existing functionality.

Testing Terminology and Concepts

Difference between Error, Defect, Bugs, Issues

Error

Mistakes found in program by programmers

Defect

Mistakes found by testers

Bugs

Actual defects accepted by developers

Issues

Difficulties in solving bugs by developers

Relationship Questions

Question Answer
If every error can be defect? No, errors may not always manifest as defects
If every error can be a bug? No, only errors that cause incorrect behavior become bugs
If every error can be issues? No, only errors that are difficult to fix become issues
If every defect can be a error? Yes, defects originate from errors
If every defect can be bug? No, only defects accepted by developers become bugs
If every defect can be issues? No, only defects difficult to resolve become issues
If every bug can be a error? Yes, bugs originate from errors
If every bug can be a defect? Yes, bugs are a subset of defects
If every bug can be issue? No, only bugs difficult to fix become issues
If every issue can be error? Yes, issues originate from errors
If every issue can be a defect? Yes, issues are a subset of defects
If every issue can be a bug? Yes, issues are a subset of bugs

Hot Fix and Production Issues

Defects found by customers in production environment are called hot fix and production issues.

Process Flow

  1. After UAT completion, product goes to production
  2. Regression test engineer deploys product
  3. Customers use the product
  4. When customers find defects, they contact customer care
  5. Customer care assigns ticket to tech support team
  6. If unresolved, ticket is transferred to production team (developers and testers)
  7. BA assigns hot fix team (developer and tester)
  8. After fixing, testing (retesting) begins

Testing Terminology Overview

Different approaches used for testing software applications when formal test cases are not predefined.

Monkey Testing

Used when time is limited and only high-priority test cases need execution.

Example: Quickly validating login, checkout, and payment features before release deadline.

Exploratory Testing

Testing without predefined test cases when testers have domain knowledge.

Example: Navigating through banking app without test plan to find unexpected errors.

Ad-hoc Testing

Quick, informal testing without documentation when tester is familiar with application.

Example: Randomly checking social media app features before release.

Component Testing

Testing performed on each individual component separately without integrating with other components. Also known as Unit Testing, Program Testing or Module Testing.

Shakedown Testing

Testing system in live environment in controlled manner to ensure all components work together as expected. Occurs after integration testing when system is considered stable.

BVA and ECP

Boundary Value Analysis (BVA) and Equivalence Class Partitioning (ECP) are techniques to identify test cases efficiently.

Equivalence Class Partitioning (ECP)

  • Divides input data into groups where all inputs behave similarly
  • Reduces total number of test cases while covering maximum functionality
  • Example: Password field (8-12 characters):
    • Valid: 8-12 characters ("password123")
    • Invalid: Less than 8 ("pass")
    • Invalid: More than 12 ("passwordislonger")

Boundary Value Analysis (BVA)

  • Focuses on testing values at edges of input ranges
  • Catches boundary-related defects
  • Example: Password field (8-12 characters):
    • Boundary: 8, 12 characters
    • Just outside: 7, 13 characters

Defect Status Details

  • New/Open: When the defect is first logged
  • Assigned: When the defect is assigned to a developer
  • In Progress: When the developer is actively working on the defect
  • Deferred: When the defect is postponed for later releases
  • Rejected/Not a Bug: When the defect is invalid or not reproducible
  • Fixed/Resolved: When the developer has implemented the fix
  • Reopened: If the defect is not resolved during retesting
  • Closed: When the defect is verified as fixed
  • Duplicate: When the defect is already reported by someone else

Traceability Matrix

Document that establishes relationships between testing process elements like requirements, test cases, and defects.

Purpose

  • Ensure all requirements are covered by test cases
  • Identify gaps in testing
  • Track test coverage

Types

  • Forward Traceability: Links test cases with requirements
  • Backward Traceability: Links test results and defects back to requirements

Test Coverage

Measures how much of the software code base is covered by existing test suites, usually expressed in percentage.

Helps evaluate testing effort and analyze root causes in critical conditions (bugs found in live environment).

Seven Principles of Testing

  1. Testing Shows the Presence of Defects: Testing identifies defects but doesn't prove their absence
  2. Exhaustive Testing Is Impossible: Focus on critical and high-risk areas
  3. Early Testing: Start testing early in SDLC to reduce cost
  4. Defect Clustering: Small number of modules contain most defects
  5. Pesticide Paradox: Regularly update test cases to maintain effectiveness
  6. Testing Is Context-Dependent: Tailor testing to project context
  7. Absence-of-Errors Fallacy: Passing tests doesn't guarantee defect-free product
