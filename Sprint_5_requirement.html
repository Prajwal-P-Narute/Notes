Sprint 5 Requirements

Description, Acceptance Criteria, and Impact

Edit Transaction

As a user, I should be able to edit an existing transaction so that I can correct mistakes or update details such as amount, category, label, or description.

Acceptance Criteria
  • User should be able to select an existing transaction and edit its fields
  • Updated values should reflect immediately in the transaction list
  • Calculations should update automatically based on the new transaction details
  • System should maintain data consistency and not duplicate records
Impact
  • Current Balance, Total Expense, and Total Income will be recalculated after edit
  • Data accuracy will improve by allowing correction of errors

Delete Transaction

As a user, I should be able to delete a transaction so that I can remove unwanted or incorrect entries.

Acceptance Criteria
  • User should be able to select a transaction and delete it
  • Deleted transaction should be removed from the transaction list
  • Calculations should update correctly after deletion
  • System should prevent accidental deletions by confirmation prompt
Impact
  • Current Balance, Total Expense, and Total Income will be recalculated after deletion
  • Database will maintain only valid transactions

Master Module for Category

As a user, I should be able to manage (add, update, delete) expense and income categories so that I can personalize the classification of my transactions.

Acceptance Criteria
  • User can add new categories under Debit or Credit type
  • User can edit the name of an existing category (including default)
  • User can delete any category unless it is currently linked to an existing transaction
  • UI should display updated category list in dropdowns

Impact
  • Transactions will reflect user-defined categories
  • Flexibility for personalization
  • Possible re-mapping needed when deleting default categories

Remove Reimbursable (Y/N) Column from UI

Remove the "Reimbursable (Y/N)" column from the UI since it is not needed, but keep backend functionality intact.

Acceptance Criteria
  • Reimbursable column should no longer be visible on transaction listing screens
  • Backend logic should continue to work
  • Existing reports or calculations dependent on reimbursable should continue working
Impact
  • Cleaner UI
  • No functional change in calculations, only visual removal

Add Label Functionality for Transactions

As a user, I want to add labels to my transactions (like "Essential", "Luxury", "Recurring") so that I can analyze my necessary and unnecessary expenses in more detail.

Acceptance Criteria
  • User should be able to create, update, and delete labels as a master list
  • While adding/editing a transaction, user should be able to assign one or more labels
  • Labels are non-mandatory
  • Labels should be displayed along with the transaction in listing screens
  • User should be able to filter transactions by label
Impact
  • New labels field will be added to transaction table
  • Users gain advanced categorization for analysis
  • Filtering and reporting logic will be extended to include labels
